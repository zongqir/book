# Writing Rubric

## Goal

让读者看完以后能回答：
- 什么时候该调用这本书
- 我最容易在哪种局面判断错
- 这本书的方法从哪一刀切进去

重点不是"写几个故事"，而是"留下最值得反复调用的场景"。

## 核心判断

好案例通常同时具备六条：
- 典型：高频、常见、易复发
- 可识别：读者一看就知道现实里会遇到
- 有错法：能看到常见误判和默认反应
- 有转折：能看到方法介入后路线怎么变
- 有边界：能分清适用条件和伪同类问题
- 可调用：能留下明确触发器

## 怎么找案例

- 优先从问题模式发现，不优先从故事内容发现。
- 候选场景先过三条硬门槛：
  高频出现 / 改法后结果变化大 / 能代表整本书。
- 一个案例默认只覆盖一个主冲突、一个关键误判、一个主要切入口。
- 候选阶段先做内部取证表，但除非用户特别要求，不要把取证表原样写进正文。

## 怎么组织案例

- 案例按调用优先级组织，不按章节顺序。
- 允许厚薄不一：
  主案例写透，辅案例补差异，提醒型案例只留触发器和边界。
- 如果多条高质量案例同时满足硬门槛，且彼此主冲突、误判、切入口都不同，不要为了控长硬压单页。
- 如果压在一页会削弱导航性和调用效率，优先拆成 `_index.md` + 分组子页。

## 可读性

案例页的信息密度比入口稿高，但不能因此写成查表页。

- 案例标题写场景，不写"案例一"——读者应该从标题就知道这是什么处境
- 不要每个案例都用完全相同的子标题结构；五个案例全是"典型错法 / 正确做法 / 反例与失效条件"三段式，读起来像数据库不像文章
- 调用信号、误判、切入口、边界这四件事都要有，但可以织进自然段落，不一定要挂子标题
- 案例之间的厚薄差异是对的：主案例三五段展开，辅案例两三段，提醒型一段搞定
- 段落之间留呼吸；不要每段都一样长、一样密

## 什么时候不要硬写

如果这本书主要提供的是概念辨析、历史脉络、世界观或框架定义，而不是可反复调用的处境处理方法，可以保留文件位，但不要硬造案例。

## 假案例清单

下面这些默认不是好案例：
- 把观点摘要写成案例
- 把行动指南写成案例
- 把漂亮故事当案例
- 把推演场景伪装成原书案例
- 把章节顺序当案例顺序

## 门禁

命中任一条，默认不过稿：

- 段首还在评论案例有多值钱，不是在写场景本身
- 案例标题换一本到别的书上也成立，说明还是抽象标题
- 读完一条仍然不知道调用信号和关键误判
- 正文更像行动指南或价值页，不像场景页

## 自检

- 这一条有没有明确处境，而不只是一个抽象观点？
- 读者看完这一条，能不能知道"我什么时候已经进入这个场景"？
- 这条是不是只覆盖一个主冲突、一个关键误判和一个主要切入口？
- 这条有没有最基本的取证支撑，而不是凭感觉留下来的？
- 这条更像案例、行动指南，还是价值页？如果不像案例，继续收。
- 这条读起来像在讲一个真实场景，还是像在填一份表格？
