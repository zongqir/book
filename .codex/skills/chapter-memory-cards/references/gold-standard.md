# 内存分析别被数字骗

内存问题最大的麻烦不是看不懂数字，而是看到数字就下结论。

free 显示内存快用完了，你急着加内存；RSS 一直涨，你觉得肯定泄漏；全局内存够用，你就不查内存了。这三个场景全是坑。

这几张卡收的都是“看起来很明显但经常判错”的点。排障时先翻一遍，少走弯路。

{{< memory-card chapter="7" title="free 不等于可用" phenomenon="free 命令显示 free 列很低，看起来快用完了。" dont="不要直接认定内存不够。Linux 会把空闲内存拿去做缓存，free 列低是正常状态。" check="看 available 列，不看 free 列。available 才是系统真正能腾出来给新进程的量。" upgrade="当 available 也持续走低、且开始触发 swap 或 OOM，才能判定内存压力。" >}}
这是最经典的误判。新手几乎都会在这里踩一脚，老手偶尔也会忘。

复习时只需记住一件事：别看 free，看 available。这一条就能挡住大多数内存误报。
{{< /memory-card >}}

{{< memory-card chapter="7" title="RSS 涨不代表泄漏" phenomenon="进程 RSS 持续增长，怀疑内存泄漏。" dont="不要只看 RSS 曲线就报泄漏。RSS 包含共享库映射、文件缓存映射等，增长可能只是正常的缓存积累。" check="对比 RSS 和 PSS，确认私有页增量；用 smaps 拆分匿名页和文件映射页的各自趋势。" upgrade="当匿名私有页持续单调增长、且进程逻辑上不该持有这么多数据时，泄漏判断才成立。" >}}
RSS 是最容易拿到的指标，但也最容易误读。用它报警可以，用它定性不行。

关键动作是拆私有页。只有匿名私有页在持续单调涨，才配叫泄漏。共享库和文件缓存部分的增长跟泄漏无关。
{{< /memory-card >}}

{{< memory-card chapter="7" title="NUMA 不均匀比总量不足更隐蔽" phenomenon="总内存充足，但某些进程延迟异常偏高。" dont="不要只看全局内存用量。NUMA 架构下，进程绑在远端节点分配内存，访问延迟会显著上升。" check="用 numastat 看各节点的本地命中率和远端分配比例。" upgrade="当远端分配占比持续偏高、且延迟和吞吐量确认受影响时，才值得调整绑定策略。" >}}
全局指标一切正常但性能就是差——NUMA 是最常被忽略的原因之一。

这张卡的价值在于提醒你查“分布”而不只查“总量”。很多人在 NUMA 机器上排了半天，最后发现只是进程绑错了节点。
{{< /memory-card >}}

## 这组卡片的共同特征

三张卡有一个共同结构：先看到一个看起来很确定的数字，然后想下一个看起来很合理的结论，但中间少了一步关键核实。

free 少的是区分 free 和 available；RSS 少的是拆私有页；NUMA 少的是查节点分布。

记住这个模式：数字本身没错，但数字和结论之间还需要一步分解才能到达。
