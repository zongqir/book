## CPU 高，先别急着说 CPU 不够

看到 CPU 利用率高就把问题压成"CPU 瓶颈"，是现场最常见的误判。利用率高背后可能是四种完全不同的处境：

- CPU 确实在高效做工，负载上来了
- 线程已经开始排队等 CPU，饱和了
- 表面很忙，但大量周期耗在等内存、缓存未命中或地址转换——看着忙其实在空转
- 容器配额、throttling、steal time、降频把真实可用算力改写了——你看到的 100% 不是真的 100%

只看利用率这一个数字，根本分不清这四件事。下面按诊断顺序一步步拆开。

## 第一眼：先看它在忙什么——user、sys、steal、iowait

拿到一台 CPU 高的机器，第一个动作不是开火焰图，而是用 `mpstat -P ALL 1` 或者 `top` 看一眼 CPU 时间的构成。

你需要分清四块时间各自占多少：

- **user%**：应用代码在跑。大部分业务负载体现在这里。
- **sys%**：内核在跑。如果 sys% 突然升高，通常不是业务负载变大了，更可能是锁竞争、上下文切换、内核态操作（网络收发、page fault、系统调用密集）在吃时间。
- **iowait%**：CPU 在等 I/O。iowait 高不代表 CPU 忙，恰恰相反——CPU 闲着，但在等磁盘或网络把数据送回来。看到 iowait 高应该去查 I/O，不是查 CPU。
- **steal%**：这台虚拟机/容器的 CPU 时间被宿主机拿走了。steal 高说明问题不在你的应用，在底层调度。

这一步的目标是做第一层分流：问题更像在应用层（user 高）、内核层（sys 高）、I/O 层（iowait 高），还是环境层（steal 高）。

同时看一眼 **每核分布**。`mpstat -P ALL` 会显示每个 CPU 核心的利用率。如果只有几个核打满、其余核闲着，问题可能是单线程瓶颈或线程绑定不均，不是整体 CPU 不够。

## CPU 在算还是在等——IPC 和 CPI 怎么看

分流之后，如果 user% 确实高，下一个问题是：CPU 在这些周期里到底干了多少活。

两台都显示 100% CPU 的机器，含义可能完全相反。一种是 CPU 大部分时间在执行指令、推进计算；另一种是 CPU 每执行一条指令都要等很久，因为数据还没从内存搬回来。表面上都是满载，实际效率差几倍。

区分这两种情况，要看硬件性能计数器。用 `perf stat -a sleep 5` 跑 5 秒，看输出里的 **instructions** 和 **cycles**，然后算 IPC（Instructions Per Cycle）：

- **IPC > 1**：每个周期执行了不止一条指令，说明 CPU 流水线吃得很饱，大部分时间在高效计算。这种情况下 CPU 高是"真忙"。
- **IPC < 0.5**：每条指令平均要花两个以上的周期才能完成，说明 CPU 大量时间在停顿——等缓存、等内存、等 TLB。这种情况下 CPU 高是"假忙"，优化方向是解决内存访问效率，加 CPU 基本没用。
- **IPC 在 0.5-1 之间**：中间地带，需要结合具体场景判断。

IPC 的反面是 CPI（Cycles Per Instruction）。CPI 高意味着每条指令花的周期多，停顿占比大。

拿到 IPC 之后你就能做一个关键判断：如果 IPC 高且 CPU 高，问题可能真的是计算量大，加 CPU 或优化算法有意义；如果 IPC 低且 CPU 高，加 CPU 基本没用，应该去查缓存未命中、NUMA 远端访问、内存带宽这些方向。

## 100% 跑在什么频率——降频陷阱

同样是 100% CPU，背后的处理器可能跑在完全不同的频率上：

- 睿频拉高到 4.5 GHz，短时间算力很猛
- 节能策略把频率压到 1.2 GHz
- 温度或功耗约束触发了降频

100% 只是"把当前这档频率吃满了"。一台被降频到 1.2 GHz 的机器报 100%，和一台跑在 4.5 GHz 的机器报 100%，实际算力差了将近 4 倍。

用 `turbostat` 可以看到当前实际频率和功耗状态。如果你发现 CPU 利用率 100% 但频率异常低，说明可能不是负载的问题，而是降频策略、温度限制或电源管理在背后改写了可用算力。这时候的优化方向不是减少负载或加 CPU，而是先解决频率问题。

## 有没有人在排队等 CPU

利用率高只能说明 CPU 忙。真正伤害延时的是饱和——新来的线程拿不到 CPU，只能排队等。

用 `vmstat 1` 看 **r 列**（运行队列长度）可以有个大致印象。但运行队列长度只是快照，不能告诉你线程到底等了多久。更精准的做法是用 `runqlat`（BCC 工具）直接看调度等待时间的分布：

```
runqlat 10 1
```

输出是一张直方图，显示线程从"可运行"到"真正拿到 CPU"等了多久。如果大部分线程等待在微秒级，CPU 没有饱和。如果出现毫秒级甚至更长的等待，说明 CPU 已经成了排队的关键路径。

这里有一个统计学陷阱：**平均值会抹平微爆发**。秒级采样看起来一切正常，不代表 200 毫秒内没发生一轮很凶的 CPU 饱和。对延时敏感的场景，必须能看到亚秒级的突刺，不能被分钟均值安慰。所以 `runqlat` 这类能看分布的工具比 `vmstat` 的单一数字更可靠。

判断逻辑：CPU 利用率高但排队不明显 → CPU 更像在正常显影工作负载；CPU 利用率高且排队延迟已经拉长 → CPU 更可能是当前瓶颈方向。

## 热点在哪——火焰图要接着缓存、迁核和 NUMA 一起看

确认 CPU 确实在消耗大量周期之后，才有必要看"具体是哪段代码最热"。这一步通常用 `perf record` 采样再生成火焰图。

但火焰图只回答"谁最热"，不自动回答"为什么热"。一个函数占 30% 的采样，可能是因为它确实在大量计算，也可能是因为它每次调用都在等缓存行从内存搬过来。这两种情况的优化方向完全不同。

所以看火焰图的时候，要同时检查几件事：

**缓存未命中**：用 `perf stat -e cache-misses,cache-references` 看缓存命中率。如果 miss 率很高（比如超过 10%），热点函数很可能在等数据，不在真正计算。优化方向是改善数据局部性、减少工作集大小。

**线程迁核**：用 `perf stat -e migrations` 看线程在 CPU 核心之间迁移的频率。迁核意味着线程从一个核搬到另一个核，之前那个核上的缓存全部失效，新核上要重新加载。频繁迁核会放大热点——代码本身没变慢，缓存被不断刷掉了。这种情况下绑核（CPU affinity）可能比改代码更有效。

**NUMA 远端访问**：多路服务器上，每个 CPU 有自己的"本地内存"。如果线程跑在 CPU 0 上但访问的内存在 CPU 1 那边，延迟会高很多。用 `numastat` 看本地/远端内存访问比例。如果远端访问占比高，优化方向是调整内存分配策略或线程绑定，不是加 CPU。

把这几层接起来，热点才从"方向感"变成"解释链"：这段代码热 + IPC 低 + 缓存未命中高 → 优化内存访问模式；这段代码热 + IPC 高 + 无明显停顿 → 优化算法本身或减少调用次数。

## 容器和云里，先排掉配额和环境失真

只要不是裸机，上面所有诊断都可能被环境因素扭曲。在容器和虚拟化环境里，要先排掉这几类失真：

**cgroup 配额和 throttling**：容器通常有 CPU 配额限制。即使宿主机 CPU 还有余量，容器用完自己的配额就会被 throttle——线程被强制暂停，等下一个周期重新分配。用 `cat /sys/fs/cgroup/cpu/cpu.stat` 看 `nr_throttled` 和 `throttled_time`。如果 throttle 次数很高，说明 CPU 不是不够，而是配额给少了。

**steal time**：在虚拟化环境里，`steal%` 高意味着宿主机把本来属于你的 CPU 时间拿给了别的虚拟机。你的应用代码没变、负载没变，但 CPU 利用率突然上升——因为同样的工作要花更多挂钟时间才能完成。steal 高的时候，优化应用代码没有意义，应该找基础设施团队解决资源争抢。

**每核分布失衡**：有些容器只被分配了 2-4 个 CPU 核心。如果应用线程数超过可用核心数，你会看到这几个核 100% 但宿主机整体不忙。这种情况下 `top` 看到的整体利用率是假的，必须看分配给容器的那几个核的实际情况。

这一步没做的话，你后面看到的利用率、热点、甚至队列延迟都可能只是失真后的结果，会把诊断方向带偏。

## 每次 CPU 排查，手里留一句分类结论

走完上面几步之后，强迫自己把结论落成一句分类，不要含糊地说"CPU 高"：

- **做工型**：CPU 确实在高效计算，IPC 高，负载上来了。优化方向：减少计算量、优化算法、加横向扩展。
- **饱和型**：线程在排队等 CPU，队列延迟已经明显拉长。优化方向：减少并发、优化调度、加 CPU 核心。
- **停顿型**：CPU 看着忙但 IPC 低，大量时间耗在缓存未命中、内存等待、NUMA 远端访问。优化方向：改善数据局部性、绑核、调整 NUMA 策略。
- **失真型**：配额、throttling、steal、降频把可用算力改写了。优化方向：调整配额、解决资源争抢、检查频率策略。

这四种情况的优化方向完全不同。分类写不出来，就说明诊断还没走完。

## CPU 分清以后，再看内存是不是也在骗人

CPU 解决的是"利用率高到底代表什么"。下一章接着解决另一个同样会骗人的表象：内存看起来快满了，到底算不算真的内存压力。内存的陷阱和 CPU 类似——数字高不等于有问题，关键是看它高在哪、高成什么样。
