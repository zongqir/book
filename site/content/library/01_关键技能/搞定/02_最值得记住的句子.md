# 十二句提醒，把开放回路从脑子里搬进可信系统

## 开放回路为什么会把人拖垮

{{< sentence no="1" quote="只要事情还悬着，它就会在后台持续占用注意力。" >}}
{{< /sentence >}}

GTD 最值钱的一点，是把压力解释成系统问题，不先解释成人格问题。

{{< sentence no="2" quote="脑子适合产生想法，不适合保存承诺。" >}}
{{< /sentence >}}

这句把上一句又推深一层。不是你记不住，而是你根本不该用脑子干这件事。

{{< sentence no="3" quote="你之所以累，不全是因为事多，也因为太多事没有被关进可靠容器。" >}}
{{< /sentence >}}

这句适合拿来对抗“我最近就是比较忙”的笼统自我解释。很多时候不是忙，是悬挂。

## 入口和理清

{{< sentence no="4" quote="先把东西收进篮子，再决定它是什么，不要一边进一边乱分。" >}}
{{< /sentence >}}

入口阶段最怕半处理。收集和判断搅在一起，系统会从第一步开始变浑。

{{< sentence no="5" quote="收集不等于处理，处理不等于执行。" >}}
{{< /sentence >}}

这句很基础，但很多系统坏掉就坏在这三个动作总被混成一个动作。

{{< sentence no="6" quote="如果一件事还没有下一步行动，它就还不是真正可执行的任务。" >}}
{{< /sentence >}}

GTD 最硬的动作就在这里。模糊任务最会制造拖延，因为它们看起来像任务，实际上没有启动点。

{{< sentence no="7" quote="项目不是一件要做的事，项目是一串尚未完成的下一步。" >}}
{{< /sentence >}}

这句很重要，因为它把“项目”从大词拉回动作链。只要不能继续分出下一步，项目就还停在名词层。

## 清单、回顾和系统信任

{{< sentence no="8" quote="能立即清掉的小事，最好的归宿通常不是清单，而是立刻完成。" >}}
{{< /sentence >}}

这句挡住了另一个高频错法：把系统当拖延缓冲区。

{{< sentence no="9" quote="等待清单不是遗忘区，而是你把球明确交出去之后的追踪区。" >}}
{{< /sentence >}}

很多人系统里最烂的一栏就是等待，因为它总被写成“以后再说”。其实它处理的是责任已经转移、但结果还没回来。

{{< sentence no="10" quote="不回顾，再漂亮的清单都会慢慢变成过期库存。" >}}
{{< /sentence >}}

没有周回顾，系统迟早失真；系统一失真，大脑就会把提醒权收回。

{{< sentence no="11" quote="当你开始信任系统，脑子才会停止不停地重复提醒同一件事。" >}}
{{< /sentence >}}

这句直指 GTD 的终点：不是效率炫技，是信任转移。

{{< sentence no="12" quote="真正轻松不是事情变少了，而是每件事都知道自己该待在哪里。" >}}
{{< /sentence >}}

这是 GTD 最容易被忽略的结果指标。系统跑顺之后，首先出现的常常不是产量暴增，而是后台安静了。
