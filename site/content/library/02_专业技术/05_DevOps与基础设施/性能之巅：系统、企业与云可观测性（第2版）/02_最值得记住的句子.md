# 《性能之巅：系统、企业与云可观测性（第2版）》最值得记住的句子

## 最值得记住的句子

{{< sentence no="1" quote="问题没说清前，不要急着找工具。" >}}
性能分析最常见的失误，不是不会用工具，而是一开始就没把问题定义清楚。

刚接手一个模糊告警时，先把“慢在哪里、慢成什么样、影响谁”写清楚，再决定看什么工具。
{{< /sentence >}}

{{< sentence no="2" quote="先把慢拆成运行、等待和排队，再谈瓶颈。" >}}
很多误判不是因为不会看指标，而是把不同性质的时间混成了一个“慢”。

只要还没分清时间到底花在执行、等待还是排队，就不要宣布瓶颈已经找到了。
{{< /sentence >}}

{{< sentence no="3" quote="高利用率不是结论，它最多只是线索。" >}}
资源域上的高数字，只能告诉你“这里值得看”，不能直接宣布根因。

CPU 高、磁盘忙、网络重传多，都只能说明“这里值得看”，还不能说明“这里就是根因”。
{{< /sentence >}}

{{< sentence no="4" quote="真正该盯住的，往往是等待、排队和饱和开始形成的地方。" >}}
真正值得追的，不是表面利用率，而是等待、排队和饱和开始形成的位置。

吞吐还没完全塌、但尾延迟已经开始变坏时，最容易先暴露出来的往往不是打满，而是排队。
{{< /sentence >}}

{{< sentence no="5" quote="先用低成本证据缩小范围，再上高成本证据挖深。" >}}
工具顺序本质上是证据成本顺序，不是功能强弱顺序。

如果方向都还没缩小，就直接上 tracing 或重型工具，很多时候只是在更昂贵地迷路。
{{< /sentence >}}

{{< sentence no="6" quote="工具不是为了证明你的直觉，而是为了替换掉它。" >}}
最危险的错法，就是先有结论，再去找工具替自己背书。

一旦你发现自己只是想让 profile、trace 或监控图替你背书，应该马上停下来，回去重新定义问题。
{{< /sentence >}}

{{< sentence no="7" quote="CPU 忙，不等于 CPU 已经成了问题。" >}}
CPU 是最容易被误判的资源域，高并不天然等于坏。

看到 CPU 高以后，要继续问：这是正常做工、假忙，还是已经出现了真正的饱和和调度等待。
{{< /sentence >}}

{{< sentence no="8" quote="内存满，不等于内存出了问题。" >}}
日常说“内存满了”时，很多时候说的只是现象，不是诊断。

缓存占用高、RSS 变大、空闲内存变少，都不自动等于内存压力。先看回收、换页、扫描和 OOM 这类更硬的信号。
{{< /sentence >}}

{{< sentence no="9" quote="文件慢，不等于磁盘慢。" >}}
I/O 路径里隔着文件系统、缓存、写回和块层，远比“盘快不快”复杂。

看到 I/O 告警时，先分逻辑 I/O 和物理 I/O，再决定是不是该怪设备。
{{< /sentence >}}

{{< sentence no="10" quote="单边网络证据，通常只够说明你看见了症状，不够说明问题在哪一边。" >}}
网络问题里最伤人的错误，就是只看单边证据就急着归因。

只看本端的重传、延迟或超时，很容易把本端、链路、对端和中间设备混成一团。网络问题里，对照关系本身就是证据。
{{< /sentence >}}

{{< sentence no="11" quote="看不全，不是停止分析的理由；它只是提醒你换成边界证据和对照证据。" >}}
在云、容器和托管环境里，看不全已经是常态，不是例外。

到了云、容器和托管环境里，很多底层细节你拿不到，但你仍然可以用版本前后差异、实例间差异、可用区差异和边界指标继续收缩范围。
{{< /sentence >}}

{{< sentence no="12" quote="看见热点，不等于拿到根因。" >}}
热点只能告诉你时间花在哪，不能自动回答“为什么这里会成为约束”。

真正要补的是：它为什么会在这里耗时，背后卡的是哪类约束。
{{< /sentence >}}

{{< sentence no="13" quote="上 tracing 要晚，上 BPF 更要晚；先把问题定义到足够窄，再动重武器。" >}}
重工具真正适合补的是事件级证据，不适合拿来替代问题定义。

真正值得上 tracing 或 BPF 的前提，不是“工具很强”，而是你已经清楚自己缺的是哪一段证据。
{{< /sentence >}}

{{< sentence no="14" quote="优化之前先算收益上限，优化之后再问为什么真的有效。" >}}
优化不能只看结果，还得同时回答两件事：值不值得做，以及为什么真的有效。

准备开优化任务和准备写复盘时，都该先估收益，再做动作，最后把因果链补齐。
{{< /sentence >}}

{{< sentence no="15" quote="一次优化如果解释不清为什么有效，它就还不是稳定的方法。" >}}
指标变好还不够，解释不清为什么有效，就还不能把它当成稳定方法。

指标变好不够，你还得知道解除的是哪类约束、靠什么证据验证、有没有副作用、下次能不能复用。
{{< /sentence >}}

{{< sentence no="16" quote="最后沉淀下来的，不该只是几张图和几个命令，而是团队更稳的观察与验证习惯。" >}}
一次排障真正该留下的，不只是这次用了什么命令，而是团队以后怎么更稳地观察和验证。

这样下次再遇到类似问题，就不用再从头走一遍弯路。
{{< /sentence >}}

## 这些句子怎么用

- 接手一个新告警时：
  记录项：
  现象：到底是谁慢了，慢到什么程度，影响了谁。
  时间类型：这次更像运行变长、等待变长，还是排队变长。
  当前线索：现在最显眼的高指标是什么，它为什么还只是线索，不是结论。
  约束候选：最像开始形成饱和的位置在哪里，下一步该补什么证据。

- 准备上工具时：
  顺序：
  第一层：系统指标、应用指标、日志、变更记录、前后对照。
  第二层：profile、热点图、分层统计。
  第三层：tracing、eBPF、重型临时采样。

- 看到 CPU、内存、I/O、网络数字难看时：
  判断项：
  CPU 高：现在只是忙，还是已经出现调度等待、run queue 积压或 steal time 证据。
  内存满：现在只是缓存吃满，还是已经出现回收抖动、swap、major fault 或 OOM。
  文件慢：现在是应用层逻辑 I/O 变慢，还是块设备层真的变慢。
  网络抖：问题只出现在本端，还是对端、链路和中间设备也有对应症状。
  云上看不全：还能不能用不同实例、不同可用区、变更前后做对照。

- 准备做优化或写复盘时：
  补充项：
  收益上限：理论上最多能省多少时间或资源，不值得就别做。
  生效机制：这次到底解除的是等待、排队，还是某个更具体的约束。
  复用条件：下次看到什么信号时，团队可以直接复用这次方法。
