# 接手一个项目之后，从对齐理解到推动决策的操作线

你接手了一个项目，或者手上的项目开始出问题。文档有，排期有，但事情不动。下面这条操作线不是帮你"管理项目"的，是帮你开始做判断的。

## 三天能跑通的最短路径

如果时间紧，只做三件事：

**第一件：和每个关键角色单独聊十五分钟。** 不是聊进度，是问三个问题——"你觉得这个项目最该担心的事是什么？""你觉得现在最不清楚的是什么？""如果三周后出问题，你觉得最可能出在哪？"

把每个人的回答记下来。你会发现，不同人对同一件事的理解差距远比你想象的大。这些差距就是项目未来出问题的种子。

**第二件：把"我们到底在做什么"写成一段话。** 不是写文档，是写一段话——100 字以内，说清楚这个项目要解决什么问题、交付什么东西、什么时候交。写完发给所有关键角色看。

如果有人回复说"不对，我理解的不是这样"——恭喜你，你在第一周就发现了一个本来要到第六周才暴露的理解偏差。

**第三件：标出当前排期里你最不信的那一行。** 不是检查整个排期表，是找出那个让你直觉不舒服的时间点。然后去找负责那部分的人，问："这个时间是怎么估的？你最担心什么？"

这三件事做完，你对项目的真实状态会比看任何文档都清楚。

## 接手第一周：收集判断材料

最短路径跑完之后，如果项目还需要更深的介入，第一周的重点是收集材料——不是收集数据，是收集判断材料。

和所有关键角色分别聊完之后，把他们的回答放在一起比对。重点看三样东西：

**哪些理解是一致的。** 这些是你可以暂时不管的。

**哪些理解有分歧但大家不知道。** 这是最危险的——两个人以为自己在做同一件事，实际上方向不同。这种分歧如果不在第一周暴露，两个月后会变成返工。

**哪些人在回避的话题。** 有人在聊天的时候明显绕开某个话题，或者用模糊的词应付。这通常说明那个方向上有问题，但还没人愿意面对。

做完这一步之后，你应该能列出三到五个"需要在接下来两周内做判断"的事项。不是任务列表，是判断列表。

## 第二到第四周：对齐、估算、嵌入判断点

**对齐：把理解差距变成具体讨论。** 第一周发现的理解分歧，不能靠群发邮件解决。拉相关的人坐下来，把分歧说清楚，当场做决定。

对齐会议的产出不是会议纪要，是决定——"我们在 X 这件事上选了 A 方案，原因是 Y，代价是 Z"。每个对齐会议如果没有留下至少一个明确决定，就白开了。

**估算：重做排期，但这次嵌入判断点。** Berkun 不要求排期精确，但要求排期诚实。重新估算时，在关键节点加入判断点：

- 第 X 天之前如果 A 没完成，说明我们低估了这一块，需要重新讨论范围
- 第 Y 天如果测试发现超过 N 个严重问题，考虑推迟发布而不是带伤上线
- 第 Z 天做一次中期检查，问自己"如果从今天重新开始，排期还会这样排吗"

这些判断点不是装饰品。它们是提前说好的"什么时候该停下来重新判断"。

**嵌入反馈回路。** 不要等到项目结束才复盘。每两周花三十分钟做一次小复盘——"过去两周哪个判断做对了？哪个判断事后看是错的？"

## 冲突出现时：三个问题拆开僵局

项目中期一定会出冲突。两个人的方案不一致、资源分配有争议、外部需求和内部计划冲突。

Berkun 处理冲突的方式不靠调解技巧，靠拆问题。把冲突双方拉在一起，问三个问题：

1. "你们对哪些部分已经有共识？"——先把同意的部分锁住，缩小战场。
2. "分歧到底在哪一个具体的点上？"——大部分冲突拆开之后，真正有争议的点比双方以为的小。
3. "如果今天必须定方向，你各自推荐什么？为什么？"——逼出具体方案，不在抽象层面打转。

三个问题问完，通常能把一场看起来很大的分歧变成一个可以做决定的小选择。然后做决定，记录决定的原因，继续走。

犹豫是最贵的。做了一个不完美的决定，两周后发现不对还能改。犹豫两周不做决定，损失的是不可逆的时间。

## 需求变更时：让代价可见

项目中途一定有人加需求。关键不是能不能加，是让代价可见。

收到新需求后做一件事：把它放进当前排期表，然后问团队——"如果做这个，我们不做什么？还是所有东西都做但延期？延期多久？"

把答案拿给提需求的人看。大部分"必须做"的需求，在看到代价之后会变成"可以下个版本做"。

如果代价看了还是要做，那就做——但正式调整排期、通知所有人、记录决定原因。不要偷偷加进去，假装什么都没变。

## 里程碑复盘：问五个问题

到了一个里程碑或项目结束时，花一个小时做复盘。不写长文档，问五个问题：

1. 哪个判断做对了？为什么当时能做对？
2. 哪个判断做错了？当时漏看了什么信号？
3. 有没有一个问题是早就有人知道、但迟迟没人说出来的？为什么？
4. 下一次做类似项目，第一周应该先做什么不一样的事？
5. 这个项目里，最该留给下一个项目经理的一条经验是什么？

第三个问题最值得花时间讨论。如果答案是"有"，说明团队的安全感不够——坏消息浮不出来。这是项目经理下一个周期最该解决的问题。

- [ ] 能用 100 字说清楚项目在做什么

- [ ] 主要理解分歧已经浮出来并做了决定
- [ ] 排期里嵌入了判断点
- [ ] 遇到冲突时用了三个问题拆分歧
- [ ] 每个新增需求都回答了"挤掉什么"

- [ ] 里程碑复盘问了五个问题
