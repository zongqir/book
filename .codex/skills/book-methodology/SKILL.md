# 方法论总结稿

05 回答的不是"这本书讲了什么"，也不是"具体怎么执行"，而是：这本书到底提供了什么可复用的方法、方法为什么必须这样组织、以及方法的边界在哪里。

如果一段内容只有原则没有路径，不进 05。如果只有步骤没有组织逻辑，不进 05。原则留 00，步骤留 04，05 只写方法本身和方法之间的关系。

## §1 命中判断

当用户要求"写方法论总结""分析这本书的方法""判断这本书有没有方法论""补齐 05"时命中。

如果用户要求写的内容更像"具体怎么做"，优先转给 `book-action-guide`。如果更像"这本书值不值得读"，优先转给 `book-why-read`。

## §2 元数据

通用必填字段遵循 `AGENTS.md` base schema。

扩展字段：

- `methodology_basis`：方法结构依据什么提炼
- `method_type`：`strong` / `medium` / `weak` / `non-methodical`
- `method_extraction_basis`：`author-stated` / `chapter-synthesis` / `cross-chapter-reconstruction`
- `method_flow_presence`：`clear` / `partial` / `weak`

`title` 写内容判断，不写"方法论总结"。`summary` 只写方法判断，不写推荐语。

证据边界默认写进 frontmatter：`source_basis`、`evidence_status`、`method_extraction_basis`、`methodology_basis`。正文默认直接写判断，不用"从现有材料看""按当前可见材料重构"这类交代工序的话替代元数据。

## §3 工作流

### NotebookLM 提问约束

当你拿不准这本书到底是不是方法型书、主方法是否存在、或 `05` 会不会把原则硬升格成方法时，先按这个 skill 的标准去问 NotebookLM。

- 先问成立性：这本书的方法强度更接近 `strong`、`medium`、`weak` 还是 `non-methodical`？
- 再问筛选性：哪些内容是真正的方法，哪些只是原则、观点或步骤？
- 最后问退出性：如果 `05` 立不住，是原书方法性本来就弱，还是 source 还不足以看清方法流？

优先问这种问题：

- 原书有没有稳定顺序、判断分叉点和适用边界，足以支撑主方法？
- 哪一个是主方法，哪些只是支撑原则或局部技巧？
- 如果只能留一个核心方法和 2 到 4 个支撑原则，最该保留什么？

不要问：`这本书的方法有哪些？`

通用判定方法见 [../references/notebooklm-slot-gating.md](../references/notebooklm-slot-gating.md)。

### 3.1 方法强度判断

动笔前必须先判断这本书是不是方法型书。四档：

- **strong**：有明确问题类型、稳定顺序、关键判断点、较清晰边界——写完整方法骨架 + 方法流
- **medium**：有主方法，但不少部分更像支撑原则——写主方法 + 关键原则
- **weak**：方法性有限，主要提供原则和结构化判断——以原则结构为主，少写方法流
- **non-methodical**：主要是观点或案例——明确写"本书方法性有限"，不硬凑骨架

判断信号：是否针对一类问题、是否有稳定顺序、是否有判断分叉点、是否有适用边界、是否能迁移到相邻场景。

没过这一步，不开始写。

这一步默认留在你的提炼过程和 frontmatter 里。不管是枚举值（strong / medium / weak）还是自然语言变体（"方法强度不算弱""方法性有限""中等强度的方法结构"），都不要写进正文。正文里要让读者通过内容自行感受方法的丰富程度，而不是看到你在给书打方法性分数。

### 3.2 四层分离

把原书内容拆成四层，再决定什么进 05：

- **观点**：作者怎么看问题——不进 05
- **原则**：面对问题时的大方向和基本约束——进 05 但标注为原则
- **方法**：针对某类问题的可复用处理路径——05 的主体
- **步骤**：一次具体执行时怎么做——留给 04

### 3.3 候选方法识别

优先找这些信号：

- 作者反复强调"不要直接……而要先……"
- 作者把一个问题拆成阶段
- 作者持续对比普通做法和更优做法
- 作者指出失败常发生在某一步
- 作者给出条件分支或顺序依赖
- 方法可以迁移到多个相邻问题

对每个候选方法做一张方法卡：解决什么问题、触发信号、核心判断、方法流、顺序为什么这样排、常见错法、边界。

### 3.4 收束

候选方法不是全部进正文。必须判断：

- 哪一个是主方法
- 哪一些只是支撑原则
- 哪一些只是局部技巧

正文优先保留能统摄全书、可迁移、顺序稳定的方法。通常收束为一个核心方法 + 二到四个支撑原则或子方法。

"主方法"是你收束时的内部标签。终稿里不要写"主方法"——直接用方法的具体名称（如"经营修正循环""四层阅读"），或省略标签直接展开内容。

### 3.5 落稿

动笔前按这些维度想清楚。它们是分析框架，不是文章骨架——最终的段落组织、H2 数量和排列顺序由这本书的内容决定：

**⚠️ 以下维度是思考工具，不是输出结构。**

它们帮你想清楚要写什么，但不决定你的 H2 标题、段落顺序或文章结构。最终结构必须从这本书的内容中自然长出来。写完后检查：如果你的 H2 标题序列和同分类下另一本书的相同槽位一样，说明你把思考维度当成了输出模板——必须重新组织。

1. 方法的底层假设或关键前提
2. 主方法是什么、解决什么问题
3. 方法流与关键顺序
4. 核心分水岭——哪个环节判断错了整套方法就失效
5. 最常见的失效方式
6. 适用边界与方法性边界
7. 一句核心方法主张

对 weak 和 non-methodical 的书，骨架压缩为：问题意识 → 核心原则 → 有限的方法倾向 → 边界。不硬凑完整七段。

不要把下面这些骨架标签直接拿去当终稿标题：

- `方法强度判断`
- `这本书在解决什么问题`
- `成立的关键前提`
- `书里真正提供了哪些方法`
- `调用信号`
- `方法流与关键顺序`
- `核心分水岭与失效方式`
- `结构关系`
- `适用边界与方法性边界`
- `核心主张`
- `主方法`
- `方法强度`

这些是你写作时的检查框，不是读者要看的标题或正文用词。终稿标题必须直接落在内容判断上。正文里也不要出现"主方法""方法强度"这类工作流术语。

过程叙述句也一律删除。下面这些都算同一家族：

- `如果把书里的方法压缩成一条主线`
- `这本书能提炼出方法`
- `从现有材料里能稳定看出的`
- `按当前可见材料重构`
- `把现有材料压成一句经营主张`
- `把全书压成一个方法`

读者不需要知道你怎么加工，也不需要知道你此刻在做"收束""压缩""重构"。

### 3.6 证据强度标注

方法论总结天然带解释成分，必须标清三层强度：

- 作者明说的框架：直接按作者强度写
- 基于全书结构的归纳：标明归纳，不假装原话
- 为帮助理解的重构表达：标明重构

默认做法：把这些边界写进 frontmatter，不写进正文。

- `source_basis` 交代材料范围
- `evidence_status` 交代证据状态
- `method_extraction_basis` 交代作者明说 / 章节归纳 / 跨章节重构
- `methodology_basis` 交代方法结构依据

正文里如果确实需要提示强度，用内容内化写法，不用工序写法。

反例：

- `从现有材料里能稳定看出的，是……`
- `按当前可见材料重构，这里的关键是……`
- `把现有材料压成一句经营主张，就是……`

正例：

- `零售经营真正反复出现的顺序是……`
- `书里反复强调的，不是先做动作，而是先修正判断。`
- `更接近作者反复强调的经营方向，是先读顾客变化，再谈门店动作。`

## §4 终检清单

### 反同质化（必查）

- [ ] H2 标题序列和同分类下已有书的同槽位不同
- [ ] 开头第一句和同分类下已有书的开头不同
- [ ] 全篇对比句式（"不是X而是Y""与其A不如B"）合计不超过 2 次
- [ ] 没有使用可以套到任何书上的万能句式作为标题或开头

### 结构完整性

- [ ] 标题是内容判断，不是"方法论总结"
- [ ] H2 标题从内容来，不是骨架壳
- [ ] H1 和 frontmatter title 一致
- [ ] 方法强度判断已经完成，且主要体现在 frontmatter 和正文写法里，不是靠 `方法强度判断` 一节硬展示
- [ ] 有主方法识别，不是候选方法平铺
- [ ] 有方法流和关键顺序
- [ ] 有核心分水岭和失效方式
- [ ] 有一句能单独站住的核心方法主张
- [ ] 有适用边界

### 边界守住

- [ ] 没有混入 04 的执行细节（步骤、完成标准、应对建议）
- [ ] 没有退化成 00 的价值清单
- [ ] 观点、原则、方法、步骤四层没有混用
- [ ] 原则没被硬升格成方法
- [ ] weak/non-methodical 的书没有硬凑完整骨架

### 证据与表达

- [ ] 作者明说 vs 归纳 vs 重构已经标清
- [ ] 没有把推断写成定论
- [ ] evidence_status 和 method_extraction_basis 准确
- [ ] 无"本页主要在讲……"等元说明句
- [ ] 无"这不是X"式负面自定义开头
- [ ] 无评价句或重要性定级句（"方法性非常强""底层假设很明确""最值得留下来的"等，不限于开场）
- [ ] 无展示骨架的话，如 `这本书在解决什么问题`、`这套方法如何如何` 这类把写作检查框直接端给读者的句子
- [ ] 无替作者解释心理、意图或关注点的句子（如"作者真正关心的是""他坚持要……"）
- [ ] 无泛指代词反复（"这套方法""整套方法"同一篇超过两次就换具体名称或省略主语）
- [ ] H2 没有描述格式或长度（"一句话""总结""小结"）
- [ ] 无"这本书"反复（超过两次就换书名、具体名称或省略）
- [ ] 无 SKILL 内部术语泄露（"主方法""方法强度""候选方法"不应出现在正文）
- [ ] 无文章自我指涉（"这页写的是""这篇总结""这里写下来的"）
- [ ] 无过程叙述句（"如果把……压缩/收束/提炼成……"）
- [ ] 无材料过渡腔（"从现有材料里能看出""按当前可见材料重构""把现有材料压成一句……"）
- [ ] 读起来舒服，没有术语堆叠或抽象概念连排

## 输出规则

- 正文 100–180 行（weak/non-methodical 可压缩到 60–100 行）
- frontmatter 通用字段 + 扩展字段齐全
- 方法性弱的书，正文长度和结构都应压缩，不要用填充内容撑长度

## 校准与例外

- 方法极其密集的书（如性能之巅），方法流可以展开到多层递归，正文上限可放宽到 220 行
- 方法性确实为 non-methodical 的书，正文主要写问题意识和核心原则，不需要硬凑方法流
- 如果 04 和 05 的边界在某本书里特别模糊，优先在 05 里写"为什么必须这样组织"，把"具体怎么做"留给 04
