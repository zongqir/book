# 一开口就散的时候，先回到这几句

## 最该反复调用的提醒

{{< sentence no="1" quote="先给答案，再给材料。" >}}
很多表达失效，不是材料不够，而是答案来得太晚。

只要对方先拿不到回应，后面的背景、过程和细节都会先变成负担。
{{< /sentence >}}

{{< sentence no="2" quote="上层先回答问题，下层再解释为什么。" >}}
一层里最怕混着放结论、原因和例子。

把层级分开以后，别人才能知道自己现在听到的是判断、理由，还是证明。
{{< /sentence >}}

{{< sentence no="3" quote="同一层的内容，必须在回答同一个问题。" >}}
列表最常见的坏法，不是点太多，而是每一点根本不是一类东西。

只要同层不在回答同一个问题，再整齐也只是视觉整齐。
{{< /sentence >}}

{{< sentence no="4" quote="能归在一起的，先归组；不能归在一起的，不要硬并列。" >}}
归组不是为了凑三点式，而是为了让人一眼看出关系。

如果三条材料只是都很重要，它们还不算一组。
{{< /sentence >}}

{{< sentence no="5" quote="顺序不是装饰，它会替你解释逻辑。" >}}
同一组材料按时间排、按结构排，还是按重要性排，给人的理解完全不同。

顺序一乱，对方会被迫自己猜你的逻辑。
{{< /sentence >}}

{{< sentence no="6" quote="写不出结论，通常说明还没想清问题。" >}}
很多人把卡壳归到表达，其实卡在判断。

当一句话始终落不下来时，先回去问自己到底在回答什么问题。
{{< /sentence >}}

{{< sentence no="7" quote="背景不是越全越好，只要够把问题托出来。" >}}
开头最容易写长的地方，就是怕对方不理解，于是把前情补得过满。

可真正有用的背景，只服务于把当前问题推出水面。
{{< /sentence >}}

{{< sentence no="8" quote="问题分析要先拆清结构，再决定从哪一刀切进去。" >}}
复杂问题最怕一上来就讨论方案。

先把问题树画清，才能知道现在缺的是判断、证据，还是选择标准。
{{< /sentence >}}

{{< sentence no="9" quote="好表达不是说得更多，而是让别人更少费力。" >}}
表达质量最后看的不是你讲了多少，而是对方抓主干用了多大代价。

能让别人少来回一次、少追问一句，结构就已经开始生效。
{{< /sentence >}}

## 这些句子最适合在哪些场景拿出来

- 写邮件写到第三段还没出现请求时，先回到第 1 句。
- 做汇报时三页都还在铺背景，先回到第 7 句。
- 列出三点以后仍然觉得哪里不对，先回到第 3 句和第 4 句。
- 讲着讲着自己也开始迷路，先回到第 2 句和第 5 句。
- 方案迟迟定不下标题，先回到第 6 句。
- 团队讨论一个复杂问题却一直围着表面现象转，先回到第 8 句。

## 真正的用法不是背下来，而是拿它们做自检

每次输出前，只要快速问几句: 我有没有先回答问题？同一层是不是同一类东西？顺序是不是在替我解释逻辑？背景有没有只保留到刚好够用？很多稿子会在发出去之前就被救回来。
