# `topic.yml` 字段口径

这个仓库里的专题，没有 `topic.yml` 就不算正式启动。

最小可用 schema：

```yaml
title: "专题名"
goal: "一句话目标"
scope:
  in:
    - "本轮明确纳入的内容"
  out:
    - "本轮明确排除的内容"
sources:
  - title: "材料名"
    path: "sources/材料文件"
    reason: "为什么纳入"
deliverables:
  - "00_规范草案.md"
  - "01_决策记录.md"
  - "02_实施清单.md"
status: "draft"
created_at: "2026-03-24"
updated_at: "2026-03-24"
question:
  core: "这次专题到底要解决什么问题？"
success_criteria:
  - "边界清楚"
  - "材料可追溯"
  - "能形成一版可执行输出"
next_step: "先整理 3 到 8 份核心材料进入 sources/"
```

字段说明：

- `title`
  - 专题显示名，通常和目录名一致。

- `goal`
  - 只写一句话，回答“这轮研究要完成什么”。
  - 不要同时塞问题背景、方法、价值判断和行动口号。

- `scope.in`
  - 当前轮次真正纳入的范围。
  - 控制在 3 到 6 条以内，避免什么都想做。

- `scope.out`
  - 当前轮次明确排除的范围。
  - 用来防止专题失控。

- `sources`
  - 先列计划纳入或已纳入的核心材料。
  - 可以先留空列表，但不要假装已经完成收集。

- `deliverables`
  - 输出物清单。
  - 项目规范类专题默认保留 `outputs/` 三件套。
  - 如果是别的专题，可以按需要调整，但文件名最好稳定。

- `status`
  - 只使用：`draft`、`active`、`review`、`archived`
  - `draft`：目标和边界已写，材料未齐
  - `active`：材料进入分析，开始形成结论
  - `review`：已有初稿，正在审阅和修正
  - `archived`：阶段目标完成，暂不继续推进

- `question.core`
  - 当前专题的核心问题。
  - 尽量写成一个真正会影响材料选择和输出设计的问题。

- `success_criteria`
  - 研究完成到什么程度才算“这一轮够了”。
  - 优先写可判断标准，不写抽象愿景。

- `next_step`
  - 写下一步最直接的动作。
  - 应该能被转成 `dida-cli` 任务，而不是一句大口号。

推荐做法：

- 信息还不全时，先写最小版本，不要因为不确定就不启动。
- 如果当前只是大模型先行分析，允许 `sources: []`，但要在其他文档里标明证据状态。
- 如果专题已经明显偏离目标，优先收紧 `scope`，不要继续堆材料。
