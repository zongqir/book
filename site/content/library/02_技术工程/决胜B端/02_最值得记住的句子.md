{{< sentence >}}
B端产品的核心不是用户体验，是业务流程的系统化表达。
{{< /sentence >}}

C端产品经理转B端后最大的惯性是继续用"用户喜不喜欢"来判断优先级。但企业客户买你的产品不是因为好用，是因为能跑通他的业务。一个流程走不通的系统，按钮再漂亮也是废品。把"体验优先"切换成"流程优先"，是B端入门的第一道关。

{{< sentence >}}
需求不是用户告诉你的，是从业务流程里挖出来的。
{{< /sentence >}}

C端靠用户访谈、问卷、数据分析来获取需求。B端的需求来源是业务流程分析——角色有几个？每个角色在流程里做什么？什么条件触发流转？异常怎么处理？客户说"我想要一个审批功能"，你得追问到"谁发起、谁审批、超时怎么办、驳回后走哪条路"才算拿到需求。

{{< sentence >}}
做B端不是做一个产品，是做一套解决方案。
{{< /sentence >}}

C端产品通常是一个独立应用。B端产品几乎不可能独立运行——它要和客户的ERP对接、和OA打通、和财务系统做数据同步。你交付的不是一个软件，是一个能嵌入客户现有系统的解决方案。不理解这一点，需求评审永远过不了。

{{< sentence >}}
B端产品上线不是终点，是实施的起点。
{{< /sentence >}}

C端产品上线了，用户下载、注册、使用，流程就闭合了。B端产品上线只是开始——后面还有部署、数据迁移、用户培训、权限配置、定制开发。很多B端产品经理在"产品上线"的那一刻觉得大功告成，然后发现客户那边用不起来，项目开始失控。

{{< sentence >}}
一个B端产品能卖出去不难，难的是客户用起来之后不找你麻烦。
{{< /sentence >}}

销售阶段靠的是演示效果和方案包装。但客户签约之后的满意度取决于实施质量。如果实施做得差，客户上线后天天报问题，续约就没了。B端产品的竞争力不是"能卖出去"，是"卖出去之后客户不想换掉你"。

{{< sentence >}}
权限设计是B端产品的隐形地基，搭错了整栋楼会歪。
{{< /sentence >}}

C端产品的权限通常很简单——注册用户和游客。B端的权限是多维度的：角色权限、数据权限、组织权限、功能权限，四个维度交叉组合。权限设计没想清楚就开始做功能，后面每加一个角色都要重构一次。这是B端产品返工率最高的领域。
