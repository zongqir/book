# 六句话把分析从堆答案拉回建假设

## 假设比数据先走一步

{{< sentence book="思考的技术" author="大前研一" >}}
不要从答案开始思考。先建立假设，再去验证。
{{< /sentence >}}

拿到商业问题，最自然的反应是搜集数据。但数据量不代表分析质量。没有假设的搜集，像是在大海里撒网——网越大，捞回来的垃圾越多。

先写一个可以被推翻的假设，再决定去找什么数据。方向比勤奋重要。

{{< sentence book="思考的技术" author="大前研一" >}}
假设不是为了猜对答案，是为了让验证有方向。
{{< /sentence >}}

很多人觉得假设就是猜。不是。假设是分析的导航工具。有了假设，你才知道该验证什么、该搜集什么数据。没有假设，你只是在堆材料。

好的假设不需要一开始就正确。它只需要足够具体，能被证实或推翻。

## 验证的硬标准

{{< sentence book="思考的技术" author="大前研一" >}}
一条有力的反证，比十条支持更能推进判断。
{{< /sentence >}}

人天然倾向于搜集支持自己判断的信息。确认偏误是人类决策里最顽固的 bug。

假设验证循环要求相反的动作：主动找反例。如果找遍了也找不到有力反证，假设暂时站得住。如果找到了，立刻修正。

验证不是法庭辩论——你不是在替自己的假设打官司，是在测试它能不能扛住冲击。

## 现场拦住办公室推演

{{< sentence book="思考的技术" author="大前研一" >}}
现场有你在办公室里看不到的东西。
{{< /sentence >}}

逻辑可以在办公室里自洽。但前提假设对不对，只有现场知道。

大前研一做咨询时反复遇到一种场景：方案在总部推演得滴水不漏，到了客户工厂一看，前提条件就不成立。数据对、逻辑对、结论却是错的。

一线数据不是为了装点报告。一线数据决定你的前提假设还能不能站住。

这句话最该在你觉得"分析已经够了"的时候拿出来。越觉得够了，越可能是在办公室里自我说服。

## 问题有结构，不只有原因

{{< sentence book="思考的技术" author="大前研一" >}}
不要急着找原因。先画出问题的结构。
{{< /sentence >}}

线性思考的本能是：找一个原因，推一个结论，出一个方案。但商业问题很少是一因一果。利润下滑背后同时有五六个变量互相牵扯。

先把变量关系画出来，找到真正的杠杆点，再决定往哪个方向用力。

这句话用在"越分析越乱"的时候最有效。越乱，越说明你在用线性方式处理一个非线性问题。

## 收束成一句可以执行的判断

{{< sentence book="思考的技术" author="大前研一" >}}
分析做完，如果不能变成一个让人愿意行动的提案，就等于没做。
{{< /sentence >}}

分析的终点不是报告，是判断。

很多人把分析做到"结论正确"就停了。但决策者看完不知道明天该做什么的分析，只是安慰剂。

把核心判断压成一句可以赌上去的话。这句话就是整个分析的交付物。如果压不出来，说明分析本身还没有到位。

## 六句话的调用场景

| 场景 | 调用哪句 |
|------|---------|
| 拿到模糊问题，不知道从哪下手 | 先建假设，再找数据 |
| 搜集了很多资料，越看越迷糊 | 假设是导航，不是猜答案 |
| 分析推进中，总觉得缺证据 | 找反证比找支持更推进判断 |
| 在办公室推演得很顺，但心里没底 | 去现场看假设还站不站得住 |
| 问题越分析越乱，理不清头绪 | 先画结构，找杠杆点 |
| 分析写完了，不知道怎么收尾 | 压成一句可执行的判断 |
