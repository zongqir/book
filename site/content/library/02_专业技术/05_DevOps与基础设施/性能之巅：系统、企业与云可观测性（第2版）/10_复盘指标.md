# 七个信号，判断排障顺序有没有进工作流

## 1. 排障起手动作

翻最近三次排障记录。第一个动作是什么？

- 直接打开熟悉的 dashboard 或跑熟悉的命令 → 方法还没进入起手
- 先写问题定义，再拆延迟类型，再跑 USE 扫描 → 方法已经进入起手

检测方式：看排障 timeline 的头五分钟。有没有明确的问题定义步骤。

## 2. on-CPU / off-CPU 区分频率

过去一个月排障时，有多少次主动做了延迟分解？

- 0 次 → 最关键的分析习惯还没建立
- 只在 CPU 相关问题上做了 → 部分内化
- 所有延迟类问题都会先拆 → 已经成为默认动作

检测方式：搜排障文档和聊天记录里 on-CPU、off-CPU 出现的频率。

## 3. USE 扫描覆盖度

上次用 USE 时，扫了几种资源？

- 只看了 CPU 和内存 → 覆盖不够，磁盘、网络、锁都该在清单里
- 逐项走了但跳过了饱和度 → USE 的价值被砍掉了一半
- 三项指标 × 所有资源都过了 → USE 在按设计运转

检测方式：看有没有 USE 检查表或等效的逐资源记录。

## 4. 工具选择依据

最近一次上 perf 或 BPF 之前，能不能说清「缺的是什么证据」？

回答如果是「别人都这么用」或「这个工具最强」，说明工具选择还停在习惯层面。Gregg 的方法要求工具选择跟着问题类型走：缺 CPU 分布用 profile，缺事件因果用 trace，缺延迟分布用直方图。

检测方式：排障记录里有没有「选这个工具是因为缺 X 类证据」的说明。

## 5. 可见性受限时的应对

云上或跨团队场景里，遇到「看不全」时做了什么？

- 停下来等更多数据，或推给别的团队 → 受限分析能力还没建立
- 主动切到边界证据：实例对比、AZ 对比、版本对比、steal time → 方法在受限环境下也在工作

检测方式：看跨团队排障记录里有没有使用对照分析或边界指标。

## 6. 优化闭环完整度

最近一次优化，before / after 记录了什么？

最小闭环：baseline 指标 → 变更内容 → 约束定位 → 验证方式 → 优化后指标 → 副作用评估。

- 只记了「延迟降了 X%」→ 闭环不完整
- 能说清解除的是什么约束、用什么指标验证 → 闭环基本完整
- 补了副作用评估和复用条件 → 达到 Gregg 要求的标准

检测方式：拉出最近的优化 PR 或复盘文档，逐项检查。

## 7. 复盘文档的可复用性

把最近一份排障复盘给一个没参与过的同事看。问两个问题：

1. 下次遇到类似问题，能不能从这份文档直接找到排查路径？
2. 文档里有没有明确的复用条件——看到什么信号时走这条路径？

如果同事的回答是「能看懂发生了什么，但不知道下次该怎么用」——说明复盘还停在记录事实层面，没有沉淀成方法。

## 三个月检查点

| 指标 | 未内化 | 部分内化 | 已内化 |
|-----|--------|---------|-------|
| 排障起手 | 直接跑工具 | 有时先定义问题 | 每次先定义 + 拆延迟 |
| 延迟分解 | 不做 | CPU 问题时做 | 所有延迟问题都做 |
| USE 覆盖 | 只看 1–2 种资源 | 扫多种但跳饱和度 | 三项 × 全资源 |
| 工具选择 | 凭习惯 | 有时说得清理由 | 每次对应证据需求 |
| 受限分析 | 停下来 | 偶尔用对照 | 系统性切换证据类型 |
| 优化闭环 | 只看结果 | 补了约束定位 | 完整 before/after 链 |
| 复盘复用 | 记录事实 | 有分析路径 | 可直接复用 |
