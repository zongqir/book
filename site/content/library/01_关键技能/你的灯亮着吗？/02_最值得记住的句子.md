# 十二句提醒，把题目从症状、责任和解法冲动里拎出来

这些句子不该被当成格言收藏。它们的用途只有一个：在你已经想开始动手时，把你往后拽半步。

## 先认现象，不急着认题目

{{< sentence no="1" quote="症状最先出现，真正的问题常常躲在症状后面。" >}}
灯亮着、客户骂了、项目延了，这些都只是入口。把入口直接命名成题目，是最常见也最贵的错误。
{{< /sentence >}}

{{< sentence no="2" quote="看到麻烦，不等于已经看到了问题。" >}}
这句话看起来重复，实际是另一层提醒。前一句讲症状和问题分层，这一句讲观察和定义之间不能自动连线。
{{< /sentence >}}

{{< sentence no="3" quote="如果解法来得太快，值得先怀疑你碰到的只是表层。" >}}
很多浅题目都给人一种“我会”的快感。真正难的问题，通常不会让人第一秒就舒服。
{{< /sentence >}}

## 题目到底是谁的

{{< sentence no="4" quote="一个问题最容易被处理错的地方，是谁都默认它该由别人解决。" >}}
这句不是在抱怨甩锅，而是在提醒：问题归属本身就是题目的一部分。归属不清，行动会迅速走偏。
{{< /sentence >}}

{{< sentence no="5" quote="你接到的问题，不一定就是你的问题。" >}}
客服承压，不等于问题属于客服；项目经理最累，不等于延期属于项目经理。最痛的人，常常只是最先挨到的人。
{{< /sentence >}}

{{< sentence no="6" quote="当一个问题牵连太多人时，先缩回你真正能负责的那一块。" >}}
边界不收，问题会无限膨胀；边界一收，真正能动的部分才会出现。
{{< /sentence >}}

## 解法不等于解决

{{< sentence no="7" quote="好解法不只要让眼前安静，还要检查它会把麻烦推到哪里去。" >}}
很多“成功处理”只是把问题搬家。投诉少了，信号也没了；流程顺了，责任更乱了；现场安静了，协作也一起没了。
{{< /sentence >}}

{{< sentence no="8" quote="把一个问题彻底消灭，可能会顺手毁掉别的东西。" >}}
这句把二阶后果的强度抬得更高一点。不是“可能有副作用”，而是“你的胜利条件本身可能太窄了”。
{{< /sentence >}}

{{< sentence no="9" quote="解决得太彻底，常常说明你把系统里别的功能也一起抹掉了。" >}}
这句特别适合拿来防“制度洁癖”。凡是特别干净、特别统一、特别一刀切的解法，都值得多问一次。
{{< /sentence >}}

## 改题、停手和放弃

{{< sentence no="10" quote="真正难的，不是想出办法，而是在投入之后承认自己接错了题。" >}}
这句直指最硬的一刀。许多人不是不会重命题，而是前面已经投入了，就不愿意回头。
{{< /sentence >}}

{{< sentence no="11" quote="不是所有问题都值得做到底。" >}}
这句和“负责到底”是有张力的。因为现实里有些题目该移交，有些该暴露，有些只该先止血。
{{< /sentence >}}

{{< sentence no="12" quote="有些问题最好的处理，不是修好，而是停在代价可接受的位置。" >}}
这不是消极，是成熟。知道什么时候收手，本身就是问题处理能力的一部分。
{{< /sentence >}}
