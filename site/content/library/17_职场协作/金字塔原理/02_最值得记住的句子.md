## 先说结论，再给理由

{{< sentence >}}
任何一篇文章的结构，都应该是一个金字塔——先给出结论，再展开支撑。
{{< /sentence >}}

动笔之前默念一遍。如果写了三段还没出现核心主张，停下来——结论还没想清楚。这句话纠正的不是写作习惯，是"边写边想"的思考惰性。

{{< sentence >}}
读者的注意力是稀缺资源。如果你不在开头告诉他你的想法，他会自己去猜——而且多半猜错。
{{< /sentence >}}

适合在写邮件、改方案时调用。你觉得"背景很重要所以先交代"，对方觉得"看了半天不知道你要什么"。注意力浪费在猜测上，远比你省掉的铺垫更贵。

## 思考本身需要结构

{{< sentence >}}
写作的过程就是思考的过程。如果写不清楚，说明还没有想清楚。
{{< /sentence >}}

卡在"不知道怎么写开头"的时候，实际卡点是论点还没成型。写不出金字塔结构，不是表达技巧不够，是思考还没完成。

{{< sentence >}}
每一层思想都必须是对下一层思想的概括总结。
{{< /sentence >}}

用来检查自己的大纲。上一级标题能不能一句话概括下面的所有内容？如果不能，说明分组有问题——要么归错了类，要么漏了一层。

## 分组的基本纪律

{{< sentence >}}
同一组里的思想必须在逻辑上属于同一范畴，而且要满足 MECE——相互独立，完全穷尽。
{{< /sentence >}}

列完三条理由之后拿这句话检查一遍：有没有哪两条说的其实是同一件事？有没有明显漏掉的？MECE 不是分类理论，是一个两秒钟的自检动作。

{{< sentence >}}
任何一组思想之间，只存在两种逻辑关系：演绎和归纳。
{{< /sentence >}}

汇报时用来定位自己到底在做什么。"因为A所以B所以C"是演绎；"A、B、C 都指向同一个结论"是归纳。分不清的时候，说明逻辑链条还没理顺。

## 开场的标准动作

{{< sentence >}}
序言的目的不是介绍新信息，而是提醒读者已经知道的事实，然后引出他还不知道的结论。
{{< /sentence >}}

写方案开头的时候调用。SCQA——情境、冲突、问题、答案——不是为了"写得漂亮"，是为了让读者在进入正文之前已经和你站在同一个起点。序言里出现了读者不知道的新概念，铺垫就失败了。

## 在什么场景下调用这些句子

写邮件前：先问自己"结论是什么"，写在第一行。写不出来就不要发。

改方案时：检查每一级标题是否概括了下面的内容。标题和内容脱节的地方，就是论证断裂的地方。

汇报前：确认自己用的是演绎还是归纳。演绎要确保每一步推理站得住，归纳要确保分组满足 MECE。

被问"你到底想说什么"的时候：回到金字塔顶端，把结论重新说一遍。如果说不出一句话的结论，说明金字塔还没搭完。
