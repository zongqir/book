# 最有价值的案例

把 `03_最有价值的案例` 当成"调用场景页"，不是故事页，也不是观点摘要页。

## 核心原则

- 这页的中心不是"怎么写案例"，而是"为什么留下这个案例"。
- 案例优先从问题模式发现，不优先从故事内容发现。
- 一个案例默认只覆盖一个主冲突、一个关键误判、一个主要切入口。
- 主案例写透，辅案例补差异，提醒型案例只留触发器和边界。
- 如果这本书不适合强写案例页，要明确退出，不要硬凑假案例。
- 案例页的信息密度比入口稿高，但节奏不能因此变成匀速平铺。案例之间该有厚薄差异，段落之间该有呼吸。

## 1. 先判定是否命中

出现下面这些情况，就按这个 skill 处理：
- 用户直接提到 `03_最有价值的案例`、`03` 文件、案例页
- 用户要新建、重写、改写、润色、审阅案例文稿
- 用户说案例写得像观点摘要、缺场景、缺边界、排序混乱、和别的槽位重复
- 任务语境明确在读书库或研究资料库里处理案例文档

下面这些任务不要落到 `03`：
- 讲为什么读这本书：去 `00_为什么读这本书.md`
- 讲具体步骤和执行顺序：去 `04_行动指南.md`
- 讲方法结构和部件关系：去 `05_方法论总结.md`
- 讲适用前提、不要硬套、失效信号和停退换判断：去 `06_适用边界与失效条件.md`
- 只想展开跨场景共性失败模式：只有材料特别厚时，再去 `07_反例与失败条件.md`

## 2. 先补元数据

通用字段与语义统一遵循仓库 `AGENTS.md`。这个槽位的扩展字段：`case_basis`、`selection_basis`、`ordering_rule`、`doc_style`。

```yaml
case_basis: "原书原则 + 高频阅读场景重构"
selection_basis: "高频场景 + 误判明显 + 纠偏后差异显著"
ordering_rule: "按调用频率排序，不按章节顺序"
doc_style: "cases"
```

同时先把这两个通用字段写对：
- `title` 不要直接写 `最有价值的案例` 或 `案例与反例`，而要把正文最核心的场景判断压成一句内容标题
- `summary` 直接概括这些案例在帮读者判断什么，不要写成 `整理了若干案例……` 这类任务说明

## 3. 先判断适不适合强写案例页

下面这些书要先谨慎：
- 主要价值在概念辨析
- 主要价值在历史脉络梳理
- 主要价值在定义和框架，而不是处境处理方法
- 没有稳定的方法介入点

如果判断"不适合强写案例页"，可以保留文件位，正文明确写暂不建议强写、更适合转向哪个槽位。

## 4. 选案例

### NotebookLM 提问约束

当你拿不准 `03` 到底有没有、案例会不会写假、或原书案例与重构母场景边界不清时，先用这个 skill 的标准去约束 NotebookLM。

- 先问成立性：原书里有没有足够多的明示案例，能支撑正式案例页？
- 再问筛选性：哪些场景最能代表整本书的方法，而不是只是故事好看？
- 最后问退出性：如果案例不足，是原书本来不靠案例立住，还是 source 还没覆盖到关键章节？

优先问这种问题：

- 哪些原书明示场景最符合"高频场景 + 误判明显 + 改法后变化大"？
- 哪些内容只能写成"重构母场景"，不能冒充原书案例？
- 如果只留 3 到 5 个案例，最该留下哪几个，哪些候选应该删掉？

不要问：`这本书有哪些故事可以写？`

通用判定方法见 [../references/notebooklm-slot-gating.md](../references/notebooklm-slot-gating.md)。

不要先问"书里有哪些故事能写"，先问：
- 这本书最常会在哪些现实局面里被调用
- 哪些局面最容易被误判
- 哪些局面一旦改法，结果变化最大
- 哪些局面最能代表整本书

默认先列 8–15 个候选场景，再筛选。候选场景先过三条硬门槛：高频出现、改法后结果变化大、能代表整本书。同时满足三条的优先保留。

每个案例的粒度：
- 一个案例只覆盖一个主冲突
- 一个案例只保留一个关键误判
- 一个案例只突出一个主要切入口

出现多个主冲突就拆；同一主冲突的不同变体做"主案例 + 辅案例"。

来源层级默认分三类：原书明示案例、基于原则重构的典型场景、为帮助调用而抽象出的母场景。不要把后两类写得像书里原封不动就在讲那个场景。

## 5. 单页还是拆分

选例完成后，再决定结构。案例密度判断是结构工具，不是选例替代品。

先根据这些特征判断案例之间的关系：
- 场景依赖：案例之间是否存在前后场景衔接
- 顺序依赖：是否必须按特定顺序阅读
- 互动依赖：案例之间是否互相对照或互补
- 失败复发：是否多条案例围绕同一个反复出现的失败模式
- 实战导向：是否需要按处境类型分组才好调用

常用判断：
- 低密度（4–6 个案例）：通常单页足够
- 中密度（6–8 个案例）：建议先分组
- 高密度（8 个以上）：优先拆成总览页 + 子页

如果经过硬筛选后，仍有多条案例同时满足三条硬门槛，且彼此主冲突不同、误判不同、切入口不同，不要为了控制篇幅硬压成单页。如果这些案例压在一页会明显削弱导航性和调用效率，优先拆成 `03_最有价值的案例/_index.md` + 分组子页。

触发拆分的理由不是书里故事多，而是高质量、可调用、彼此差异明确的案例足够多。只有在案例之间确实存在清晰分组时才拆，不要为了显得完整而机械建目录。

总览页最小输出约定：
- 这本书最该被调用的 3 类局面
- 建议先看的 2 个案例
- 案例分组逻辑
- 不适合从案例页进入的情况
- 遇到什么信号时，应从案例页切去 `04_行动指南.md`
- 主案例 / 辅案例 / 提醒型案例导航

如果采用目录拆分：
- `_index.md` 负责说明为什么要拆、分组逻辑、建议先看的 2 个案例、适合从哪类局面进入
- 子页再展开主案例、辅案例和提醒型案例

## 6. 成稿

先看 [writing-rubric.md](./references/writing-rubric.md) 和 [gold-standard.md](./references/gold-standard.md)。

**⚠️ 以下四件事是分析维度，不是固定子标题。**

每个案例应该根据自己的场景决定怎么组织内容。不同的书、不同的案例，段落结构可以完全不同。写完后检查：如果所有案例都用了完全相同的子标题序列，说明你在套模板。

每个案例至少回答这四件事（但不要求每个案例都用完全相同的子标题）：
- 调用信号：你怎么知道自己已经进入这个场景
- 典型误判：人最容易怎么判断错
- 方法介入：这本书的方法从哪一刀切进去
- 适用边界：什么情况下适用，什么情况下别乱套

主案例写透，辅案例和提醒型案例可以减重，但至少保留调用信号和一句关键边界。

**关于呈现方式的几条硬规则：**

- 案例标题写场景判断，不写"案例一""案例二"
- 不要每个案例都用完全相同的子标题结构——如果五个案例全是"典型错法 / 正确做法 / 反例与失效条件"三段式，读起来像在查表，不像在读文章
- 主案例可以展开三五段，辅案例可以两三段写完，提醒型案例甚至可以一段搞定。厚薄不同是对的
- 每个案例的边界可以织进正文，不一定要单独挂一个"适用边界"小标题

如果拿不准结构和文风，直接参照 [gold-standard.md](./references/gold-standard.md)。

## 7. 终检清单

### 反同质化（必查）

- [ ] H2 标题序列和同分类下已有书的同槽位不同
- [ ] 开头第一句和同分类下已有书的开头不同
- [ ] 全篇对比句式（"不是X而是Y""与其A不如B"）合计不超过 2 次
- [ ] 没有使用可以套到任何书上的万能句式作为标题或开头


提交前逐条过一遍：

**核心**
读者读完以后，能回答"什么时候该调用这本书"和"我最容易在哪种局面判断错"

**元数据**
- [ ] `title` 是内容判断，不是 `最有价值的案例` 或 `案例与反例`
- [ ] `summary` 直接写场景判断，不是 `整理了若干案例……` 这类任务说明
- [ ] `created_at` / `updated_at` 是完整带时区的 ISO 时间，不是纯日期
- [ ] `evidence_status` 与实际来源一致；没有贴源就写 `llm-draft`
- [ ] `case_basis`、`selection_basis`、`ordering_rule` 已填写

**案例质量**
- [ ] 每个案例至少包含调用信号、误判、方法切入口和边界
- [ ] 每个案例只覆盖一个主冲突，没有把多个场景压在同一条里
- [ ] 案例来源层级标清楚：原书明示、重构场景、母场景
- [ ] 没有把观点摘要、行动指南或漂亮故事当案例
- [ ] 没有把推演场景伪装成原书案例

**结构**
- [ ] 案例按调用优先级组织，不按章节顺序
- [ ] 案例标题写场景判断，不写"案例一""案例二"
- [ ] 不是所有案例都用完全相同的子标题结构
- [ ] 主案例厚、辅案例薄、提醒型案例更薄——有明显的厚薄差异
- [ ] 正文 H1 与 frontmatter `title` 一致

**槽位边界**
- [ ] 没有滑成入口稿（→ 00）或行动指南（→ 04）
- [ ] 没有混入方法论结构（→ 05）或边界条件（→ 06）
- [ ] 如果有反例内容，确认是案例的边界补充，不是独立的失败模式分析（→ 07）

**文风**
- [ ] 读起来像在讲真实场景，不像在填表
- [ ] 没有连续出现"这本书告诉我们""作者指出"这类转述腔
- [ ] 段落有呼吸——不是每段都一样长、一样密
- [ ] 标题和段首句读起来像自然中文，不像模板填空
- [ ] 开头没有反向自我定义，结尾没有给文章写使用说明

**长度**
- [ ] 单页案例稿正文在 100–180 行之间；低于 100 行说明案例没展开，超过 180 行检查是否可以拆分或是否混入了别的槽位内容

## 输出规则

- 案例标题写场景判断，开头直接进入内容
- 每个案例都是一个可识别的处境，不是一个抽象观点
- 案例之间有厚薄差异，段落之间有呼吸
- 文风保持平实对话感，避免教案腔和分析提纲腔
- 改现有稿子时，优先保留已经成立的有效案例，不做机械式全量重写
- 正文典型长度 100–180 行；低于 100 行通常说明场景没有展开，超过 180 行要检查是否需要拆分

## 校准与例外

- 如果内容主要在回答"这本书值不值得读"，它更像 `00_为什么读这本书.md`。
- 如果内容主要在回答"具体该怎么做"，它更像 `04_行动指南.md`。
- 如果只有观点，没有处境，不是案例。
- 如果只有步骤，没有误判，也不是案例。
- 如果材料不足以支撑高质量案例，宁可收缩数量，也不要硬凑满版。
