# 专题启动质量评分标准

## topic.yml 的 5 条硬标准

1. **goal 只写一件事**：不要把目标、方法、价值判断塞进同一句。"形成一版可落地的 RAG 规范"合格，"深入研究 RAG 并总结最佳实践和落地经验"不合格。
2. **scope 够窄**：`scope.in` 控制在 3 到 6 条。超过 6 条说明没收敛，不如拆成两个专题。
3. **scope.out 不是摆设**：至少有一条明确排除项，是你真的可能会做但这轮故意不做的事。
4. **deliverables 可验收**：每个输出物是一个具体文件，不是"一份报告""若干结论"。
5. **next_step 是最短动作**：能被直接转成一条 dida-cli 任务。"全面调研"不合格，"整理 3 份材料进入 sources/"合格。

## 常见坏味道

| 坏味道 | 为什么差 | 改法 |
|--------|----------|------|
| `goal: 深入研究 X` | 没有完成标准 | 改成可验收的产出，如"形成一版 X 规范" |
| `scope.in` 超过 6 条 | 专题太散 | 砍到 3-6 条，或拆成子专题 |
| `scope.out` 为空 | 没有边界意识 | 至少写一条你可能会做但这轮不做的事 |
| `next_step: 开始研究` | 口号，不是动作 | 写成可执行的最短一步 |
| `topic.yml` 全是模板占位 | 没有收紧 | 脚手架生成后立刻逐字段改 |

## 首轮动作检查

- 动作是否够短够具体，不是大口号
- 每条是否绑定了文件路径
- 待验证判断是否能转成下一轮任务
- 有没有过度前置——在材料不足时就试图产出结论

## README 检查

- 第一段是否交代了痛点和目标
- 有没有写成"本专题旨在……"的套话
- 纳入和排除是否与 `topic.yml` 一致
