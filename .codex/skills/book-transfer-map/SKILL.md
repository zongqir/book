# 迁移地图稿

09 回答的不是"这本书怎么用"，也不是"这套方法好不好"，而是：方法的底层机制离开了这本书的话题之后，还能在哪里起作用、具体怎么起作用。

一篇好的迁移地图让读者产生跳跃感——从"这本书讲的是 X"跳到"我正在做的 Y 也可以用这个思路"。

## §1 命中判断

当用户要求"写迁移地图""分析这本书的方法还能用在哪里""补齐 09"时命中。

如果用户要求写的内容更像"具体怎么用这本书"，优先转给 `book-action-guide`。如果更像"方法结构和组织逻辑"，优先转给 `book-methodology`。如果更像"适用边界"，优先转给 `book-boundaries-and-failure-modes`。

**和 04 行动指南的核心区分：**

| 09 迁移地图 | 04 行动指南 |
|---|---|
| 方法搬到书外其他场景 | 怎么读/用这本书本身 |
| "同样的底层机制，在 Y 场景里叫什么" | "第一步做什么，第二步做什么" |
| 跨场景应用 | 读书闭环执行 |

如果一段在描述"如何用这本书做某件事"，那是行动指南的内容，不属于迁移地图。

## §2 元数据

通用必填字段遵循 `AGENTS.md` base schema。

扩展字段：

- `transfer_basis`：迁移判断的依据，例如"核心方法抽取 + 工作场景映射"或"底层机制提炼 + 类比场景识别"
- `audience`：目标读者
- `confidence`：判断置信度

`title` 写迁移判断的内容，不写"迁移地图"。`summary` 只写迁移主线，不写推荐语。

输出文件：`09_迁移地图.md`，放在对应单书目录下。模板见 [09_迁移地图.template.md](./assets/09_迁移地图.template.md)。

## §3 工作流

### 3.1 先抽取底层可迁移机制

不要从书的表层内容直接类比。动笔前回答三个问题：

- 这套方法的核心动作是什么？（最抽象的那一层）
- 这套方法解决的是什么类型的问题？（不是这本书的具体主题，是更底层的问题类型）
- 同样的动作放到书外场景里，叫什么？

先把这个底层机制写出来，再去找迁移目标。从底层机制出发找到的迁移目标，比从书中例子直接类比出来的更稳。

### 3.2 识别 4 到 6 个迁移目标

迁移目标要具体到读者能对号入座。不接受"所有需要管理的事情""各种学习场景"这类过泛的目标。

每个迁移目标必须能回答：

- 用了这套方法以后，这件事具体会怎么做？
- 不用这套方法的话，通常会怎么做？（对比才能看出迁移的价值）
- 需要做哪些调整才能让方法在这个场景里运作？

### 3.3 写清楚每个迁移场景

先看 [writing-rubric.md](./references/writing-rubric.md)。

每个迁移场景需要三件事（不必用三段式结构，但内容要到位）：

1. 目标场景是什么
2. 方法的哪个底层机制可以迁移过来
3. 迁移后具体怎么做，和没用这套方法时有什么不同

终稿 4 到 6 个场景。超过 6 个通常意味着每个都点到为止。少于 3 个说明底层机制还没抽够。

### 3.4 检查有没有混入行动指南内容

逐段检查。如果某个"迁移场景"的描述实际上是"如何用这本书做……"，那是行动指南的内容。要么移到行动指南，要么改写成真正的跨场景迁移。

### 3.5 落稿

H2 标题从迁移目标来，不从骨架来。不要用"迁移场景一""迁移场景二"当标题。

正文开头先用一小段写清底层可迁移机制，然后逐场景展开。

不要在正文写这些：

- `迁移场景 1 / 迁移场景 2`
- `底层可迁移机制`（作为标题）
- `迁移目标`
- "可以广泛应用于各种场景"

这些是写作检查框，不是读者要看的标题或正文。

## §4 终检清单

### 结构完整性

- [ ] 有底层机制提炼，不是直接从表层例子类比
- [ ] 有 4 到 6 个具体迁移场景
- [ ] 每个场景说清了"用了之后具体怎么不同"
- [ ] 标题是迁移判断内容，不是"迁移地图"
- [ ] H2 标题从迁移目标来，不是骨架壳

### 边界守住

- [ ] 没有混入 04 的执行细节（"怎么用这本书"）
- [ ] 没有退化成 05 的方法结构分析
- [ ] 迁移目标不泛（没有"所有需要规划的事情"）
- [ ] 迁移后的做法具体到能执行，不是只说"可以参考"

### 证据与表达

- [ ] evidence_status 准确
- [ ] 没有把推断写成定论
- [ ] 无"本页主要在讲……"等元说明句
- [ ] 无评价句（"非常值得迁移""特别有价值"）
- [ ] 无泛指代词反复（"这套方法"超过两次就换具体名称）
- [ ] 无 SKILL 内部术语泄露
- [ ] 无过程叙述句
- [ ] 读起来舒服，有跳跃感

## 输出规则

- 正文 80–130 行
- frontmatter 通用字段 + 扩展字段齐全
- "可以应用到所有地方"不是迁移地图，是在回避这道题
