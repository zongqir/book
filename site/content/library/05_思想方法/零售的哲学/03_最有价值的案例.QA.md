# 案例页曾暂缓，现已补齐的内部说明

2026-03-28 更新：这份判断已经不是当前状态。`03_最有价值的案例.md` 已基于原书 EPUB 与 NotebookLM 首轮贴源核对补齐；本页保留，单纯为了记录此前为什么一度不该急着写。

这次没补 `03_最有价值的案例.md`，不是因为书中缺案例，反而更像是因为书中很可能案例不少，而当前材料还不够支撑把它们写准。

现有文稿已经比较稳地收住了一条主线：先看顾客变化，再提出经营假设，再用门店动作和系统反馈持续修正。这个方法骨架可以先写出来，所以 `00`、`04`、`05` 能先成立。可案例页要回答的是另一件事：到底哪些现实局面最能代表整本书，哪些是原书明说的典型案例，哪些只是根据经营主张重构出来的母场景。

这一步比方法重构更吃来源层级。现在手里的材料还不足以稳定地区分：

- 原书明确展开过的案例
- 根据目录和简介可以合理推回去的场景
- 只是因为这套方法看起来适用，所以大模型顺手补出来的局面

这三层如果没分开，案例页就很容易写假。

## 现在硬写，最容易假在哪里

最危险的假，不是写错一个细节，而是把“合理推演”写成“原书案例”。

《零售的哲学》很容易让人顺手写出几个看上去很对的零售场景：早餐客流变晚、晚间即食更快、顾客更愿意为稳定品质和省事付钱。它们当然可能贴近作者会讲的东西，但在当前证据状态下，还不能把这种重构直接当成“这本书最有价值的案例”。

另一种更隐蔽的偏差，是把方法页里的动作闭环反过来伪装成案例。看起来像是在举场景，实际只是把“变化信号 -> 假设 -> 小验证”套进几个常见零售局面。这样的稿子能读，但不够真，也不利于以后贴源回看。

## 为什么这次先留方法，不先留案例

因为当前更稳的是“这本书反复在强调哪种经营顺序”，而不是“书里哪几个案例最该优先留下”。

方法骨架允许先根据目录主线和有限抽样做结构重构，只要在元数据里把证据边界写清楚，就还能成立。案例页则更依赖对场景来源、误判转折和作者到底拿哪一类局面说服读者的把握。这个层面如果还没稳，就宁可先不留。

## 什么条件下值得重开

下面这些条件里，至少满足两条，再开 `03` 会更稳：

- 拿到更多正文样章、目录细目或可贴源材料，能确定原书到底反复展开哪些场景
- 能把“原书明示案例”“重构母场景”“迁移场景”分开标注，不混写
- 已经完成一轮贴源核对，知道哪些案例真的是作者拿来支撑全书主张的
- 能回答“为什么留下这几个，不是另外几个”，而不是只因为它们顺手好写

到那时再写案例页，留下来的才更像原书最值钱的调用场景，而不是当前整理阶段的合理猜测。
