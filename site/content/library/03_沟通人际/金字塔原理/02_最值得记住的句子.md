# 写之前先问自己在回答什么问题

## 结论先行

{{< sentence no="1" quote="先回答问题，再给理由。" >}}
邮件第三段还没出现请求时，回到这一句。答案晚出场一段，对方的耐心就少一截。
{{< /sentence >}}

{{< sentence no="2" quote="写不出结论，通常说明还没想清问题。" >}}
卡在措辞上的人少，卡在判断上的人多。先放下键盘，回去确认自己到底在回答什么。
{{< /sentence >}}

## 归组与同层纪律

{{< sentence no="3" quote="同一组内容必须在回答同一个问题。" >}}
三条并列的建议，一条讲现象、一条讲原因、一条讲动作——这不是三条建议，是三种东西混在一层。
{{< /sentence >}}

{{< sentence no="4" quote="能归在一起的先归组，不能归在一起的不要硬并列。" >}}
三条材料只是"都很重要"，不算一组。归组的判断标准是关系，不是重要性。
{{< /sentence >}}

## 排序、开场与问题拆解

{{< sentence no="5" quote="顺序本身就在替你解释逻辑。" >}}
按时间排，对方理解成因果链。按重要性排，对方理解成优先级。顺序选错，对方会拿错框架来接你的内容。
{{< /sentence >}}

{{< sentence no="6" quote="背景只需要够把问题托出来。" >}}
开头容易写长，因为怕对方不理解。有用的背景只做一件事：让当前必须回答的问题浮出水面。问题一旦站住，回答就该出现。
{{< /sentence >}}

{{< sentence no="7" quote="复杂问题先拆清结构，再决定从哪里切进去。" >}}
讨论一开始就争方案，通常是因为没人先把问题按现象、原因、约束拆开。结构拆清以后，讨论会自动聚焦到具体分支上。
{{< /sentence >}}
