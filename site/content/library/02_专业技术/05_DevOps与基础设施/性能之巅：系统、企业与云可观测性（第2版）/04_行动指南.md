# 《性能之巅》行动指南

## 接手告警：头五分钟做什么

1. **写下问题定义**。谁报的、什么慢了、慢到什么程度、影响范围多大。不要跳过这步直接开终端。

2. **拆延迟类型**。当前的「慢」更像 on-CPU 执行变长，还是 off-CPU 等待变长，还是排队堆积？这决定后面看什么。

3. **跑 USE 扫描**。对 CPU、内存、磁盘、网络逐项检查利用率、饱和度、错误。目标是 5 分钟内排除大部分无关资源。

4. **查变更记录**。最近的部署、配置变更、流量变化。时间相关性是最廉价的证据。

头五分钟的产出：一份问题定义 + 一份已排除列表 + 一两个值得深入的方向。

## 定位瓶颈：逐层 drill-down

USE 扫描指向了某个资源域之后，下一步是在这个域内继续分层。

**CPU 方向**：

- 先看 mpstat 确认 user、system 还是 iowait
- 饱和度看 run queue 深度和调度延迟
- 需要 on-CPU 分布时上 perf + 火焰图
- 需要 off-CPU 分析时看 offcputime 或 off-CPU 火焰图

**内存方向**：

- 先看 free / vmstat 确认回收压力
- 关注 page scan rate、swap in/out、major fault
- 进程级看 RSS 增长趋势和 page fault 类型
- 需要分配热点时用 BPF 工具（stackcount on kmalloc）

**磁盘方向**：

- 先分逻辑 I/O（VFS / 文件系统层）和物理 I/O（块设备层）
- iostat 看设备利用率和服务时间
- 延迟分布看 biolatency
- 需要文件级归因时用 fileslower 或 ext4slower

**网络方向**：

- 先确认问题在本端还是对端，两端对照
- 看重传、RTT、连接状态分布
- 内核网络栈延迟用 tcplife、tcpretrans
- 需要抓包时 tcpdump 按需过滤

每个方向的原则相同：先用计数器排除，再用 profile 定位，最后用 trace 补事件级细节。

## 决定优化前：先算值不值得

定位到瓶颈之后，先回答三个问题再动手。

**收益上限是多少？** 瓶颈占总延迟 5%，优化到零也只快 5%。Amdahl 定律在性能优化里是硬约束。

**改动的风险和代价？** 调内核参数、改锁策略、换 I/O 调度器——每个动作都有副作用。收益不够大时，风险可能不值得承担。

**有没有更简单的解法？** 加缓存、调超时、限流、分流——有时候一个配置变更比重构代码更经济。

## 优化落地：before / after 怎么做

优化前后的对比，不能只看「快了多少」。最小记录集：

| 项目 | 内容 |
|------|------|
| baseline | 优化前的关键指标快照（延迟分布、吞吐、错误率） |
| 变更内容 | 具体改了什么，一句话 |
| 约束定位 | 解除的是什么约束（等待、排队、锁、I/O、调度……） |
| 验证方式 | 用什么指标确认因果关系 |
| after | 优化后同一组指标 |
| 副作用 | 有没有其他指标变差 |

缺了任何一项，复盘时就说不清「为什么有效」。

## 写复盘：沉淀方法

复盘的目标是让团队下次不用从头来。聚焦三件事：

1. **分析路径**。从告警到根因，经过了哪些判断节点、排除了哪些方向、在哪里切换了工具。

2. **关键证据**。哪条证据锁定了瓶颈。计数器、profile、trace、对照——说清类型和来源。

3. **复用条件**。下次看到什么信号时，可以直接走同一条路径。

不要写成流水账。读者是三个月后遇到类似问题的同事，需要的是路径和证据。

## 一页清单

| 阶段 | 核心动作 | 产出 |
|------|---------|------|
| 接手告警 | 定义问题 → 拆延迟 → USE 扫描 → 查变更 | 问题定义 + 排除列表 + 方向 |
| 定位瓶颈 | drill-down：计数器 → profile → trace | 约束定位 + 证据 |
| 决定优化 | 算收益上限 → 评估风险 → 选最简路径 | 优化决策 |
| 落地验证 | before / after 全量记录 | 对比表 |
| 写复盘 | 路径 + 证据 + 复用条件 | 可复用方法 |
