# Repository Guidelines

## 目标

这个文件只负责两件事：

1. 告诉大模型这个仓库现在的真实结构是什么
2. 告诉大模型什么时候必须调用哪个 skill

具体文稿怎么写、每个槽位的结构和方法论，以下沉到对应 skill 为准，不在 `AGENTS.md` 里重复规定。

## 当前仓库结构

这是一个以读书拆解和站点内容生产为中心的仓库，不是应用代码仓库。

当前实际工作重心在：

- `site/content/library/`：正式资料库内容
- `site/static/diagrams/`：站点直接引用的图文件、导图与关系图产物
- `00_inbox/`：新收集的原始资料
- `.codex/skills/`：项目级 skills
- `scripts/`：内容校验、站点启动和辅助脚本

当前资料库分类主要在：

- `site/content/library/01_关键技能/`
- `site/content/library/02_专业技术/`
- `site/content/library/03_沟通人际/`
- `site/content/library/04_理财/`
- `site/content/library/05_思想方法/`

默认围绕 `site/content/library/` 工作，不要在根目录再维护一套平行正文树。

如果用户明确在做展示层、页面收录、图文件引用或评论开关相关任务，也直接修改：

- `site/layouts/`
- `site/static/`
- `site/config.toml`

注意：当前仓库里没有现成的 `90_专题研究/` 目录。凡是涉及专题模板、`topic.yml`、`sources/`、`outputs/` 的规则，只能在用户明确要求新建专题时启用，不能当作默认前提。

## 默认工作方式

当用户要求：

- 看看某本书
- 按项目约定拆解一本书
- 补齐某本书的固定文稿
- 改写资料库某篇正式文章

默认目标是直接修改 `site/content/library/` 下对应目录与文稿，而不是只给建议。

默认顺序：

1. 先确认目标书或目录在 `site/content/library/` 下的实际位置
2. 再确认现有单书目录是否已经存在
3. 缺什么补什么，不重复造轮子
4. **差异化发现**（详见下方“动笔前必须完成的差异化诊断”）
5. 优先调用匹配的项目级 skill
6. 明确区分模型初稿与贴源稿

### 动笔前必须完成的差异化诊断

每本书在动笔写任何槽位之前，必须先完成差异化诊断。这一步的产出直接决定后面每篇文稿的结构、切入角度和表达节奏。

**诊断三问（必须逐条回答，不能跳过）：**

1. **这本书的论证方式是什么？** 是案例驱动、原则演绎、经验归纳、对话辩论、历史叙事、还是实验报告？论证方式决定了文稿应该怎么组织——案例驱动的书适合从场景切入，原则演绎的书适合从判断切入，历史叙事的书适合从时间线切入
2. **同分类下已有的书用了什么结构和切入角度？** 读一遍同目录下已有书的 H2 标题序列和首段。这本书必须和它们不一样——标题序列不同、首段切入不同、段落节奏不同
3. **这本书自己的语气和节奏是什么？** 作者写得克制还是激烈？论证密还是松？干货集中在哪里？文稿的语气应该跟着书走，不是所有书都写成同一种“成熟读者的理性分析”

**诊断结果如何使用：**

- 诊断结果写入当前工作上下文，每个槽位 skill 在生成前必须读取
- 如果诊断发现“这本书的结构和已有某本书太像”，必须主动换一种组织方式
- 诊断不需要写成文件，但必须在生成过程中实际执行并影响产出

**Codex 批量生成时的额外要求：**

批量生成多本书时，每本书的差异化诊断必须包含对“前面已经生成的书”的结构回顾。具体做法：在生成第 N 本书之前，先列出前 N-1 本书的开头句式、H2 标题序列、段落节奏特征，然后确保第 N 本书和它们都不同。

### 高歧义请求默认路由

下面这些说法默认都算“高歧义请求”：

- 看看某本书
- 分析某本书
- 聊聊某本书
- 先做一下某本书
- 解读某本书

处理这类请求时，必须额外遵守：

1. 不要先靠模型习惯脑补成某个槽位
2. 先看同分类下是否已经有稳定单书样例，再复用它的文件集合
3. 如果分类和默认落点已经足够清楚，就直接按项目默认套件开工，不要反过来把“高歧义”当成停工理由
4. 只有在“分类也不清楚”或“同类样例没有形成稳定默认落点”时，才停在“确认目录 + 明确拟落槽位”这一步，不要直接写正文

### 默认核心套件

当用户说“按项目约定拆解”“按项目默认来”“直接解读这本书”，且分类已清楚时，默认直接补这组槽位，不再逐个确认：

`00` `02` `03` `04` `05` `06` `10`

文件名与 skill 映射见下方“单书固定槽位”。

- 核心套件优先于模型临场猜测；用户没逐条点名，也不退回“只给建议”
- 如果当前书不适合公开留某个槽位，按过程材料规则留同名 `.QA.md`，写清原因
- 对原则密度和论证密度都明显高的书（思想经典、理论类、论战类、强判断型文本），除核心套件外再补 `01` 和 `08`


## 全局统一约束

这些约束跨 skill 生效：

- 默认输出中文，除非用户明确要求别的语言
- 默认以一个认真读过并反复消化这本书的成熟读者视角写作，但不要假装掌握未提供的细节；所有判断都要受当前材料范围和 `evidence_status` 约束
- `_index.md` 是目录入口页，不是正式正文；默认只写短 frontmatter，不写正文
- 单书正式文稿不能只保留 `title/created_at/updated_at/status`
- 正式文稿默认至少包含：`title`、`book_title`、`created_at`、`updated_at`、`status`、`summary`、`intent`、`source_basis`、`evidence_status`
- 如果仓库已有稳定字段、稳定文件名或稳定页面形态，优先兼容，不要为了一次任务硬换
- 固定槽位文稿动笔前先做槽位边界判断：先确认“这页只回答什么”，再写正文
- 固定槽位文稿动笔前，先确认目标目录、目标文件和触发的 skill；说不清时，不要直接写正文
- 固定槽位文稿默认先补齐 frontmatter 再写正文；通用字段遵循 `AGENTS.md`，skill 只补槽位专属字段
- 新建或改写需要上线的文稿时，`created_at` 和 `updated_at` 默认写“昨天”的带时区完整时间，不要写今天，避免 UTC 构建把页面判成 future page
- `status` 表示流程状态，`evidence_status` 表示证据状态，`visibility` 表示展示状态；三者不要混用
- 只基于模型先验、目录、正文抽样或未核对材料时，明确写 `evidence_status: llm-draft`
- 如果当前不是贴源稿，就把边界写进元数据；不要在正文里假装已有严格来源，也不要把模型推演伪装成原书逐字内容
- 需要保留但不应该进入默认展示流的过程稿，使用 `visibility: internal`
- `04_行动指南.md` 和 `05_方法论总结.md` 不要写成互相重复的变体
- 改写现有稿子时，优先保留已经成立的有效段落，不要为了“重写过”全量推倒
- 如果任务已有对应 skill，优先使用 skill，不要只靠 `AGENTS.md` 自行发挥

### frontmatter 基座真源

`AGENTS.md` 是单书文稿 frontmatter 的唯一通用真源。skill 内的 schema 只写“扩展字段”，不重复通用字段。

单书文稿通用必填字段（base schema）：

- `title`
- `book_title`
- `created_at`
- `updated_at`
- `status`
- `summary`
- `intent`
- `source_basis`
- `evidence_status`

常用可选字段（跨槽位通用，按需保留）：

- `author`
- `audience`
- `confidence`
- `visibility`
- `differentiation_notes`


`differentiation_notes` 用法：在动笔前差异化诊断完成后，把“这本书和同分类已有书的核心差异”写进这个字段。它是跨槽位复用的，同一本书的所有槽位共享同一条。
字段语义统一约束：

- `status`：流程状态
- `evidence_status`：证据状态
- `visibility`：展示状态

时间字段统一口径：

- 面向上线文稿，`created_at` / `updated_at` 默认写“昨天”的带时区完整时间（例如 `2026-03-26T10:00:00+08:00`）
- 模板中可用占位符，但落成稿必须符合上面的时间口径

公共 example 格式（base schema）：

```yaml
---
title: "文稿标题"
book_title: "书名"
created_at: "2026-03-26T10:00:00+08:00"
updated_at: "2026-03-26T10:00:00+08:00"
status: "draft"
summary: "写一句只对这本书成立的话——把书名换掉如果还成立，说明太泛"
intent: "写一句只对这本书成立的话——每本书不同，禁止套用万能句式"
source_basis: "目录 + 正文抽样 + 现有材料"
evidence_status: "llm-draft"
---
```

各 skill 在 SKILL.md 中定义扩展字段时，只写槽位专属字段，不重复上面的公共字段。

## 全局表达质量要求

这组要求跨 skill 生效。核心目标只有一个：让读者读起来舒服。

读起来舒服是所有文稿能被读下去的前提。如果读者觉得累、觉得闷、觉得像在完成任务，内容再正确也留不下来。下面的规则都服务于这个前提。

**四条顶层原则：**

1. **直接写判断、结论、动作。** 读者打开这页是来获取内容的，跳过一切“我接下来要给你看什么”
2. **用具体对象替代泛指。** 写书名、场景名、具体判断。“这本书”“真正重要的”“很多人”都是偷懒
3. **段落长短交替，每段只说一件事。** 连续三段以上同样长度，读者就开始走神
4. **语义保真优先于文采。** 不为了顺口增删限定词、因果关系和结论强度

下面的具体规则都服务于这四条。

**通俗易懂硬指标**

下面四条是"读起来不累"的底线，所有正式文稿默认遵守：

- **句子短**：单句不超过 35 字。需要更长时（比如前后对比），用分号或破折号拆成子句，不要逗号一路连到底。整篇超过 35 字的句子不超过 15%
- **词近人**：术语首次出现时，同句或下一句必须用日常话让它自解释。同一句里不要塞超过 3 个专业词汇；拆成两句，每句消化一个概念
- **段有气**：不允许连续 3 段以上都在同一长度区间（都 3-5 行或都 1-2 行）。关键判断后插一个 1-2 行短段收住，再展开下一个
- **入口低**：每个小节的首句必须是不查资料、不翻前文就能懂的普通中文。首句就是术语或抽象判断的，前面补一句场景或问题把读者接进来

这四条同时也是所有 skill 终检清单的默认检查项，不必在每个 skill 里重复列出。

项目级常见写作反模式统一见 [`.codex/references/writing-anti-patterns.md`](./.codex/references/writing-anti-patterns.md)。

`AGENTS.md` 只保留跨全仓库都成立的类别级原则；某个槽位的具体禁例、替换方式和自检项，优先下沉到对应 skill。

**节奏与呼吸**

- 段落要有呼吸感。允许一两句话单独成段收住一个判断，不要每段都写成三四句的均匀块
- 长段和短段交替出现，读起来才不闷；如果连续三段以上长度差不多，主动拆一拆
- 每段只承载一个核心判断；一段塞了多个判断的，拆开
- 不要把太多信息塞进一个句子。逗号接逗号的长复句，拆成两三个短句通常更好读
- 每句都要有信息推进，不要为了“像文章”去填充过渡句
- 不要先用一句抽象概括，再用下一句具体展开同一件事。如果具体的那句已经说清了，抽象的那句就是多余的——直接删掉，让具体内容自己开口

**语气与文风**

- 默认语气像在跟一个认真的朋友说话——不像在写报告，不像在交作业，不像在发公众号
- 通俗易懂优先于花哨表达；能用常用词就不用生硬术语；必须保留术语时，让上下文足以自解释
- 不要生造复合词。把两个常用词硬拼成一个口语里不会出现的组合（如“成功部件”“经营纪律”），读起来会卡。判断标准：如果这个词你跟朋友说话时说不出口，就拆成自然的表达
- 避免口号化、套话化、模板腔表达；句子要服务理解，不服务“显得有力量”
- 文风一定要避免明显的 AI 味。不要堆砌“首先/其次/最后”“值得注意的是”“需要指出的是”“不难发现”这类自动写稿腔；不要写得像在完成一份标准答案
- 不要用评价句或重要性定级句领起段落。“方法性非常强”“最大的贡献是”“非常值得一读”“最底层的”“一切都建立在它上面”——它们打等级而不解释内容。先展开内容，读者自己判断重要性
- 不要用空心评价句给内容搭台。`最耐读的一条线`、`真正能穿出具体历史处境的`、`最怕的错法`、`真正厉害的地方` 看起来像在推进，实际只是在拖时间。直接把那条线、那个误判、那个判断写出来
- 不要用“很多人”做跳板。“很多人这样想”“常见的误读是”“经常被忽略的是”——先搭一个“大家错了”的梯子再翻转的写法，用一两次可以，反复用就变成结构拐棍。如果判断有力量，不需要先描述别人的错误
- 对比句式（“不是X而是Y”“与其A不如B”“看起来C其实D”）单篇合计不超过 2 次。超过就是在用句式推进文章，说明判断本身还不够有力。写完后搜一遍，多的改掉
- 写完后搜索“其实”“真正”“经常被”。每个命中位置，试着删掉这个词——如果后面的判断依然成立，说明这个词是空转词，直接删掉
- 避免泛指代词反复出现。“这本书”“这套方法”“整套体系”“这本书的核心”同一篇文章里出现超过两次就是偷懒；换成书名、具体名称（“四层阅读”“检视-分析这条线”）、省略主语，或直接进入内容
- 不把卷次、章节号、槽位号、目录代号当高频句首主语。`卷一`、`第一章`、`03`、`05` 这类词默认只用于定位，不用于承载正文核心判断
- 正文能省主语就省；需要指代时，优先用完整对象或自然说法，如 `《毛泽东选集（第一卷）》`、`这一卷文章`，不要长期重复用编号简称顶在句首
- 避免 `X里最值得……的是`、`X真正……的地方`、`这也是为什么 X……` 这类“对象指代 + 评价壳”句式。它们先讲评、后交内容，读起来像批注，不像正文
- 段首句优先直接下判断，不先给内容搭评价台子

**内容优先于解释**

- 少解释“你正在写什么”，多直接写。不铺导游词（“这章真正要回答的是……”“本书接下来会……”）、条件式导游词（“如果你读完……应该还在”）、过程叙述（“如果把……压缩成……”）；直接给判断和结论
- 小标题说内容本身，不说结构角色（“本章定位”“问题驱动”）或格式（“一句话”“总结”“概述”）
- 不让文章谈论自己。“这页写的是”“这篇总结”“当前这篇”都是自我指涉；不给文章写使用说明；不用“这不是 X”做反向定义。好内容不需要 README
- 不把导购分群和加工动作写成正文重心。“适合谁”“更适合哪种人”“把……压成……”一旦盖过内容，说明判断还没写出来
- 不把写作过程暴露给读者。工序免责声明（“下面保留的是基于……提炼的……”）放 frontmatter；筛选过程融入相关段落；写作内部术语（“主方法”“方法强度”“调用信号”）用具体内容替代
- 不在正文中逐条标注元信息。“证据强度：作者明说”“来源：第三章”这类表单式标注破坏阅读节奏；元信息放 frontmatter
- 如果写了卡、清单、模板、地图，至少展开最小字段或最小步骤，不要只报产物名

**精度与边界**

- 语义保真优先于文采；禁止为了“更顺口”增删关键限定词、因果关系和结论强度
- 文风可以自然，但不能拿“更像人话”当借口偷掉限定条件、范围边界和不确定性说明
- 任何涉及证据强度的表述，不得因润色被“升格”为更确定结论
- 改写旧稿时先做“保真改写”，再做“节奏优化”；不要一次性重写到失去原稿含义


**跨书多样性（反同质化硬约束）**

同质化是这个仓库当前最严重的质量问题。如果两本书的同一槽位文稿可以互相替换而读者不会发现，说明每篇文稿都没有真正服务于它所讨论的那本书。下面的规则直接针对同质化的根源。

结构多样性：
- 同分类下任意两本书的同一槽位，H2 标题序列不得相同或高度相似。如果写完发现标题序列和已有书一样，必须重新组织
- 同一槽位不存在“标准 H2 序列”。skill 骨架里的分析维度是思考工具，不是输出结构。最终的 H2 标题必须从这本书的内容中自然长出来
- 每篇文稿的开头第一句必须和同分类下已有书的开头不同。不允许出现“如果你……这本书值得……”或任何可以套到另一本书上的万能开头

句式多样性：
- 任何句式结构（包括“A，B；C，D”这种对仗句、“先X再Y”这种顺序句、“与其X不如Y”这种对比句），单篇出现超过 3 次就是句式依赖。一旦发现，拆掉多余的，换成其他表达方式
- 这条规则的执行方式是正面的：写完一篇后，扫一遍自己用了哪些重复句式，把重复的改掉

语气多样性：
- 语气跟着书走。写克制的学术书用克制的语气，写激烈的论战书用有力的语气，写实操手册用干脆的语气。不要所有书都写成“理性、平和、成熟的分析者”
- summary 和 intent 字段的写法必须反映这本书的特点。“帮助读者完成……判断或动作”是模板——替换成这本书实际能帮读者做的具体事情

## 过程材料规则

这些规则跨 NotebookLM、章节卡片、QA 补充页等流程共同生效：

- QA、审查结果、候选问题、筛选建议这类内容默认视为过程材料，不直接当正式正文展示
- 如果用户明确要保留 QA 补充页，优先命名成同名 `.QA.md`；
- 如果单书固定槽位这次明确选择“不留”或“暂不补”，默认也要留同名 `.QA.md` 作为内部诊断页，避免事后只看到缺口，看不到为什么没留
- 这类“缺失槽位诊断”页默认写明 4 件事：当前为什么不留、主要证据缺口或结构原因、现在硬写最容易写假在哪里、后续出现什么材料或信号时值得重开
- 缺失槽位诊断页默认加 `visibility: internal`，不要进入博客默认展示流
- 过程材料默认也要可读，但不要写成“候选稿说明”或“审稿记录”
- 不把当前草稿上传为 NotebookLM source；只把书籍、文章、原始资料作为 source
- 只吸收真增量：能补判断、误判、边界、证据、最小动作的内容才回写正文

## Skill 优先原则

凡是任务已经有明确对应的项目级 skill，就必须优先调用对应 skill。

这里的顺序不要搞反：

1. 先由 `AGENTS.md` 判断任务属于哪个产物、哪个槽位
2. 再调用对应 skill
3. skill 负责“这个槽位怎么写”，不负责替代 `AGENTS.md` 决定“这次到底该写哪个槽位”

如果模型已经先选了 skill，才回头替当前任务发明文件名或槽位，按违规处理。

### 单书固定槽位

当用户要求“按项目约定拆解”“补齐某本书文稿”“重写某个固定槽位”时，按下面映射调用：

- `00_为什么读这本书.md`：`book-why-read`
- `01_核心原则.md`：`book-principle-extractor`
- `02_最值得记住的句子.md`：`book-key-sentences`
- `03_最有价值的案例.md`：`book-best-cases`
- `04_行动指南.md`：`book-action-guide`
- `05_方法论总结.md`：`book-methodology`
- `06_适用边界与失效条件.md`：`book-boundaries-and-failure-modes`
- `07_反例与失败条件.md`：默认不再单独作为固定槽位；边界与失败模式默认合为一篇，使用 `book-boundaries-and-failure-modes`
- `08_论证链.md`：`book-evidence-chain`
- `09_迁移地图.md`：`book-transfer-map`
- `10_复盘指标.md`：`book-review-metrics`

补充约束：

- 上面这组映射才是当前仓库默认固定槽位；未经用户明确要求，不要自行扩写一套新的固定槽位集合
- 如果同分类下已有稳定样例，新的单书目录默认复用样例中的文件集合，不要因为“这个 skill 也存在”就主动补一个样例里没有的文件

### 高歧义动词的默认解释

这些词最容易让模型按自己的习惯乱路由，默认按下面理解：

- `分析某本书`：先做目录定位、样例比对和槽位判断；如果分类和默认套件已清楚，直接进入核心套件，不再停在拟落点
- `解读某本书`：默认按当前分类的项目套件落稿；思想类书默认额外判断是否补 `01_核心原则.md` 与 `08_论证链.md`
- `按项目约定拆解一本书`：按当前仓库固定槽位与对应 skill 补齐，不自行发明新槽位
- `补齐某本书文稿`：只补当前仓库已存在或已明确映射的固定槽位
- `逐章拆`：走 `engineering-book-chapter-breakdown`
- `贴源分析`、`原文核对`、`书里具体怎么说`：走 `notebooklm`

### 单书目录之外的常见任务

当任务类型符合下面描述时，也必须优先调用对应 skill：

- 用户要求逐章拆解一本书：`engineering-book-chapter-breakdown`。默认对工程、技术、系统类书籍最有效；非技术书逐章时把"工程判断"替换为该书的核心价值类型。逐章和单书固定槽位（00-10）独立并存：槽位做跨章综合，逐章做按章说透
- 用户要求基于原书做贴源核对、交叉比对、回写证据状态，或要求用 NotebookLM 反复刷正文、批量问答：`notebooklm`
- 用户要求生成单独的章节记忆卡片页：`chapter-memory-cards`
- 用户要求把研究推进成 dida-cli 任务闭环：`dida-research-closure`
- 用户要求初始化一个新研究专题：`topic-bootstrap`
- 用户要求提取一本书的核心原则层、区分观点与原则、或需要一组不读偏的判断准则：`book-principle-extractor`
- 用户明确说现有正文“读起来累”“太绷”“太满”“像 AI”“像总结”“像说明书”，或要求降低模板感、拆长句、换词换句、补情绪露出、增加留白时：`writing-de-fatigue`。这个 skill 负责二次重写可读性，不替代槽位判断；默认先由槽位 skill 写对边界，再用它降阅读疲劳

如果一个任务明显命中多个 skill，选择最小但足够覆盖的 skill 组合，不要把所有 skill 都挂上。

## 两阶段工作流与工具边界

单书文稿默认分两个阶段：

**第一阶段：模型先验生成。** 调用对应的 slot skill（00-10）或逐章 skill，基于模型先验、目录、抽样和现有材料生成初稿。产出标记 `evidence_status: llm-draft`。这一阶段不需要 NotebookLM。大部分书的大部分槽位在这一阶段就能出合格初稿。

补充约束：

- 对高公共认知、经典化、解释资源极厚的书，如果当前任务是结构化解读、方法提炼、场景重写或资料库整理，而不是逐字核引、版本辨析、考据争议或“原书具体怎么说”，默认停在第一阶段即可，不把接 NotebookLM 当成机械补课动作
- 这类书包括但不限于：长期被反复讨论的经典政治文本、思想经典、兵法经典、哲学经典、管理经典。判断重点不是“书老不老”，而是“公共解释资源是否已经极厚，模型先验是否足以支撑一版诚实的 `llm-draft`”
- 遇到这类书时，优先把边界写清楚，而不是为了形式完整强行贴源；只要没有伪装成逐字核对稿，`llm-draft` 可以长期成立

**第二阶段：贴源核对。** 当需要和原书交叉核对时，调用 `notebooklm` skill 做逐轮精修。满足以下任一条件时进入：

- 用户明确要求贴源、核对原文或“和书对一下”
- 需要把 `llm-draft` 升级为 `notebooklm-checked`
- 需要回答“书里具体怎么说”
- 需要多书比较或交叉验证
- 当前书虽属经典，但这次任务已经进入逐字核引、措辞强度、篇目顺序、版本差异、出处核对或争议性表述判定

第二阶段的第一轮问题优先从文章所属槽位 skill 的“NotebookLM 提问约束”里取——那些问题按槽位特性设计，比通用问法更精准。后续轮次用 notebooklm skill 的通用迭代流程。

`llm-draft` 是一个诚实的状态，不是一个需要消除的缺陷。不是所有文稿都需要第二阶段。

**dida-cli** 不是每次写单书文稿都必须接入，只在这些情况下优先考虑：

- 用户明确要推进长期研究
- 同一问题需要多轮阅读、整理、输出、审阅
- 明显出现“只收集不推进”的情况

## 常用检查命令

- `Get-ChildItem -Recurse -Depth 2`：查看目录层级
- `rg "关键词" .`：搜索书名、文稿或重复内容
- `rg -n "这本书|这套方法|整套体系|卷一|卷二|卷三|卷四|第一章|第二章|最值得|真正|一直在拆|另一个很硬的提醒|留下来的" site/content/library/目标目录`：提交前抓正文里的泛指代词、卷次主语和评价壳句式
- `python scripts/check-frontmatter.py`：检查 `site/content/library/` 下文稿元数据
- `./scripts/start-site.ps1`：启动本地站点预览


## 最低校验要求

提交前至少确认：

- 文件放在 `site/content/library/` 下的正确分类目录
- 所有正式文稿都有完整 frontmatter
- `_index.md` 没被写成正文
- 正文已跑一轮反模式搜索，至少检查 `这本书`、`这套方法`、`整套体系`、`卷一/卷二/...`、`第一章/第二章/...`、`最值得`、`真正`、`一直在拆`、`另一个很硬的提醒`、`留下来的` 这类泛指代词、卷次主语和评价壳句式
- 单篇正文里，`这本书`、`这套方法`、`整套体系`、`卷一/卷二/...`、`第一章/第二章/...` 这类对象指代如果在正文中高频充当句首主语，或合计明显超过 2 次，默认视为未过稿；先改成具体判断、具体对象或直接省主语
- `status`、`evidence_status`、`visibility` 没有混用
- 过程稿是否已经正确标成 `visibility: internal`
- 如果有明确暂不补的固定槽位，是否已经留下同名 `.QA.md` 诊断页解释原因
- 站点直接引用的图文件是否已经放到 `site/static/diagrams/`
- 如果已有对应 skill，这次确实优先使用了对应 skill
- **跨书同质化检查**：和同分类下已有书的同槽位文稿对比——开头句式是否雷同、H2 标题序列是否雷同、段落节奏是否雷同。任一项雷同则必须修改后再提交

如果任务涉及贴源，再额外确认：

- 模型推断和贴源结论已经区分开
- 来源边界写清楚，没有把未核对判断冒充成定论
