# 用几句话记住创业里最容易犯的错

这些句子不是摘抄来的名言。

它们是给正在做产品决策的人准备的刹车片。

在排需求、定版本、做宣发之前，先过一遍这几条提醒。

## 先把每句话当成一次自检

不要一次性背下所有句子。

遇到犹豫不决的产品选择时，挑一两句最扎心的对照一下。

如果答案让你不舒服，往往说明团队已经偏离了实验思维。

## 句子和场景

{{< sentence >}}
创业的任务不是做出产品，而是找出可持续的业务。
{{< /sentence >}}

如果讨论一直围着功能和界面打转，可以问一句：

这个决定会让我们更接近可持续业务，还是只会多一条发布记录。

{{< sentence >}}
每一轮迭代先写下假设，再决定要做什么功能。
{{< /sentence >}}

当天的站会没有任何人提到"我们在验证什么"，就停下来补这一行。

没有假设的迭代，只是在堆工作量。

{{< sentence >}}
最小可行产品只为学习服务，不为体面服务。
{{< /sentence >}}

如果团队为了美观和完整性一再推迟上线，提醒自己：

现在更需要的是暴露风险，而不是展示质感。

{{< sentence >}}
没有可被证伪的指标，就算不上一次实验。
{{< /sentence >}}

当你发现任何结果都能被解释成"还不错"，说明指标选错了。

真正的实验要允许结论是"这条路走不通"。

{{< sentence >}}
坚持的是愿景，路径要敢于根据数据转向。
{{< /sentence >}}

愿景可以多年不变，执行路径要在每次实验后被审视。

如果团队只会把失败归因于"还不够努力"，就已经在拒绝学习。

{{< sentence >}}
增长前先证明留存，规模前先证明单点模型。
{{< /sentence >}}

新增用户再多，如果第二周就看不到人影，扩张只是在放大漏洞。

先把一小撮用户服务好，再去追求体量。

{{< sentence >}}
所有大赌注都应该被拆成一串小实验。
{{< /sentence >}}

当一个决策的失败后果大到让人不敢动手时，把它拆成几个可回头的尝试。

每一段都要有清晰的停下点。

{{< sentence >}}
团队的燃料是学习速度，而不是发布频率。
{{< /sentence >}}

每次上线后，如果没有任何新判断被写下来，只是多了一次版本号。

真正的进展，是你能更快地指出下一步该试验什么。
