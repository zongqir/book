# 适用边界与失效条件稿

06 回答的不是"这本书讲了什么"，也不是"具体怎么做"，而是：这套方法在哪里最有力、在哪里会坏、坏到什么程度该换。

一篇完整的 06 要帮读者形成一条误用判断链，而不是只记住几条分散的警告。

边界和失败模式默认合为一篇。不再拆成 06 + 07 两个文件。

## §1 命中判断

当用户要求"写适用边界""分析失败条件""判断这本书的方法什么时候该用什么时候不该用""补齐 06"时命中。

如果用户要求写的内容更像"具体怎么做"，优先转给 `book-action-guide`。如果更像"方法结构和组织逻辑"，优先转给 `book-methodology`。如果更像"值不值得读"，优先转给 `book-why-read`。

## §2 元数据

通用必填字段遵循 `AGENTS.md` base schema。

扩展字段：

- `guardrail_basis`：边界判断的依据来源

`title` 写内容判断，不写"适用边界"或"适用边界与失效条件"。`summary` 只写边界判断，不写推荐语。

## §3 工作流

### NotebookLM 提问约束

当你怀疑 `06` 可能会写成空泛警告、材料不够支撑失效链，或不确定原书是否真的给出了边界时，先按这个 skill 的标准去问 NotebookLM。

- 先问成立性：原书里有没有足够材料支撑"适用前提 + 不适用情境 + 失效方式 + 停退换信号"这条判断链？
- 再问筛选性：哪些前提、误用和失效最值得优先留下，哪些只是泛泛提醒？
- 最后问退出性：如果 06 写不稳，是原书边界本来就薄，还是当前 source 还没覆盖到方法边界最密的部分？

优先问这种问题：

- 作者明说了哪些适用范围和前提？
- 哪些相似情境看起来能套，其实不该硬套？
- 就算适用，最容易在哪一步走偏，出现哪些信号该停、该退、该换？

不要问：`这本书有哪些边界？`

通用判定方法见 [../references/notebooklm-slot-gating.md](../references/notebooklm-slot-gating.md)。

### 3.1 收集候选判断

动笔前先拉平这 4 类内容：

- **适用场景**：什么问题类型、什么阶段、什么前提下最值得调用
- **不适用场景**：哪些看起来相似但不该硬套的情境
- **失效方式**：就算场景对了，最容易在哪一步走偏
- **停退换信号**：出现什么迹象说明该停、该退、该换方法

不要边想边写，先收集再筛选。

### 3.2 筛选优先级

候选判断不是全部进正文。优先保留：

- 最容易被过度泛化的边界
- 最容易被忽略的前提
- 最危险、最常复发的失效方式
- 最能帮助读者及时止损的信号

砍掉显而易见的（"消遣阅读不适用"如果对这本书太明显就不值得占篇幅）和过于理论化的边界。

### 3.3 落稿

动笔前按这四个维度想清楚。它们是分析框架，不是文章骨架——最终的段落组织由这本书的内容决定：

**⚠️ 以下维度是思考工具，不是输出结构。**

它们帮你想清楚要写什么，但不决定你的 H2 标题、段落顺序或文章结构。最终结构必须从这本书的内容中自然长出来。写完后检查：如果你的 H2 标题序列和同分类下另一本书的相同槽位一样，说明你把思考维度当成了输出模板——必须重新组织。

1. 什么问题和阶段下最值得用，成立需要什么前提
2. 哪些相似情境不该硬套
3. 就算适用，最容易在哪一步走偏
4. 出现什么信号说明方法已经在空转

四段不是四个独立清单，而是一条连续判断链。前一段的结论自然导向下一段的问题。

不要把下面这些骨架标签直接拿去当终稿标题：

- `适用场景`
- `不适用场景`
- `失效方式`
- `停退换信号`
- `适用边界`
- `反例与失败条件`
- `关键判断的证据状态`

这些是你写作时的检查框，不是读者要看的标题。

也不要把整篇写成"条件表 + 注意事项表 + 纠偏表"。每一段优先用叙述推进，必要时再用短列表补判断点。

### 3.4 证据标注

边界判断天然带推测成分。标清三层：

- 作者明说的适用范围：直接按作者强度写
- 基于全书结构的归纳：标明归纳
- 为帮助理解的重构：标明重构

## §4 终检清单

### 反同质化（必查）

- [ ] H2 标题序列和同分类下已有书的同槽位不同
- [ ] 开头第一句和同分类下已有书的开头不同
- [ ] 全篇对比句式（"不是X而是Y""与其A不如B"）合计不超过 2 次
- [ ] 没有使用可以套到任何书上的万能句式作为标题或开头

### 判断链完整性

- [ ] 有明确的"什么问题下最值得用"
- [ ] 有至少一段"哪些相似情境不该硬套"
- [ ] 失效方式具体到读者能对号入座
- [ ] 有可操作的停退换信号
- [ ] 四段形成连续判断链，不是四个独立清单

### 结构与标题

- [ ] 标题是内容判断，不是"适用边界"或"适用边界与失效条件"
- [ ] H2 标题从内容来，不是骨架壳
- [ ] H1 和 frontmatter title 一致
- [ ] 没有拆成 06 + 07 两个文件

### 边界守住

- [ ] 没有混入 04 的执行细节
- [ ] 没有混入 05 的方法结构
- [ ] 没有退化成 00 的推荐语
- [ ] 没有把"这个场景不该用"和"这个场景能用但会走偏"混成一堆

### 证据与表达

- [ ] 作者明说 vs 归纳 vs 重构已经标清
- [ ] 没有把推断写成定论
- [ ] evidence_status 准确
- [ ] 无评价句或重要性定级句
- [ ] 无泛指代词反复（"这本书"超过两次就换）
- [ ] 无 SKILL 内部术语泄露
- [ ] 无文章自我指涉
- [ ] 无过程叙述句
- [ ] 无生造复合词
- [ ] 无先抽象概括再具体展开同一件事
- [ ] 读起来舒服

## 输出规则

- 正文 80–130 行
- frontmatter 通用字段 + 扩展字段齐全
- 默认一篇文件，不拆分

## 校准与例外

- 方法极其密集且边界本身很复杂的书（如性能之巅），正文上限可放宽到 160 行
- 方法性极弱的书，06 可以压缩到 50 行以内，只写核心前提和最危险的误用
