# 《性能之巅》最有价值的案例

## CPU 不高，请求就是慢

团队最常踩的坑之一：CPU 利用率正常，就下结论说不是性能问题。

Gregg 在多个章节里用这类场景示范延迟分解。关键动作：先拆 on-CPU 和 off-CPU。结果经常是 on-CPU 只占请求延迟的 20%–30%。剩下的时间花在等锁、等 I/O、等调度或等下游响应。

排查方向应该从「哪个资源忙」转到「线程在哪里等」。off-CPU 分析（off-CPU flame graph、调度延迟统计）比 CPU profile 更能定位瓶颈。

利用率正常不等于没问题。先拆时间去向。

## 流量上来就塌：饱和度先于利用率

平时系统跑得好好的。流量一上来，尾延迟拉长，吞吐反而下降。

直觉反应是「容量不够，扩容」。但 Gregg 的分析路径先问另一个问题：哪个资源先饱和了？

饱和度和利用率是两件事。CPU 利用率 70% 可以正常运转；但如果 run queue 深度已经到了 8，调度延迟在飙升，利用率还没到 100% 时系统就已经塌了。

USE 方法在这种场景下特别有效。逐资源检查饱和度，找到第一个开始排队的瓶颈，后面的优化才有方向。盲目扩容经常只是把瓶颈挪到了下一个资源。

## 火焰图给了热点，改了还是慢

工具熟练度不低的团队容易踩这个坑。

火焰图显示某个函数占 CPU 时间 35%。团队优化了这个函数，CPU 占比降到 15%。但端到端延迟几乎没变。

原因：热点是 on-CPU 视角。如果请求的延迟主要来自 off-CPU 等待（下游调用、磁盘 I/O），优化 CPU 热点对整体延迟贡献极小。

Gregg 反复提醒：从热点到根因，中间隔着工作负载分析。要回答「为什么这里会成为约束」，需要把工具结果放回请求全路径里看。

这个案例适合拿来做团队校准：上次优化时，有没有先确认热点和延迟瓶颈是同一件事？

## 云上尾延迟突然变差

实例内指标正常。应用日志没报错。但 P99 延迟从 50ms 跳到了 200ms。

传统排障思路依赖完整视角。到这里就卡住了——云上拿不到宿主机数据，看不见邻居噪声，摸不到底层网络。

Gregg 在第二版里专门处理这类场景。核心动作：切换到边界证据。

- **steal time**：虚拟机被宿主机抢了多少 CPU
- **实例间对比**：同一服务不同实例延迟是否分化
- **可用区对比**：问题是单 AZ 还是跨 AZ
- **版本前后对比**：部署变更和延迟变化的时间相关性

单独看，每条证据都不够硬。组合起来，能把范围从「整个系统」缩小到「某个 AZ 的某类实例在某次部署后」。

可见性残缺时，用对照和边界证据继续推进。

## 优化生效了，但复盘写不出来

线上延迟降了 40%。团队庆祝。写复盘时发现没人能回答四个问题：

1. 解除的是什么约束？
2. 用什么证据确认因果关系？
3. 有没有副作用？
4. 下次类似问题能直接复用吗？

Gregg 把「未解释的成功」视为方法论上的未完成状态。指标变好是必要条件，因果解释才让一次优化变成可复用的方法。

复盘时要补的最小闭环：before/after 数据 → 约束定位 → 解除机制 → 副作用评估 → 复用条件。缺了这条链，下次遇到相似问题，团队还是从零开始。
