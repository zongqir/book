# 十二句提醒，拦住你在错误题目上的投入

不要当格言收藏。它们只有一个用途：在你已经想开始动手时，把你往后拽半步。

## 症状不是题目

{{< sentence >}}
症状最先出现，问题常常躲在症状后面。
{{< /sentence >}}

灯亮着、客户骂了、项目延了——这些都只是入口。把入口直接命名成题目，是最常见也最贵的错误。

{{< sentence >}}
如果解法来得太快，值得先怀疑你碰到的只是表层。
{{< /sentence >}}

很多浅题目都给人一种"我会"的快感。碰到难题的第一秒通常不会舒服。

{{< sentence >}}
看到麻烦，不等于已经看到了问题。
{{< /sentence >}}

观察和定义之间不能自动连线。你能描述一个现象，不等于你已经命名了正确的题目。

## 归属判断错了，行动就全偏了

{{< sentence >}}
一个问题最容易被处理错的地方，是谁都默认它该由别人解决。
{{< /sentence >}}

不是在抱怨甩锅。问题归属本身就是题目的一部分，归属不清，行动会迅速走偏。

{{< sentence >}}
你接到的问题，不一定就是你的问题。
{{< /sentence >}}

客服承压不等于问题属于客服。项目经理最累不等于延期属于项目经理。最痛的人常常只是最先挨到的人。

{{< sentence >}}
当一个问题牵连太多人时，先缩回你能负责的那一块。
{{< /sentence >}}

边界不收，问题会无限膨胀。边界一收，能动的部分才会出现。

## 解法自带副作用

{{< sentence >}}
好解法不只要让眼前安静，还要检查它会把麻烦推到哪里。
{{< /sentence >}}

投诉少了，信号也没了。流程顺了，责任更乱了。很多"成功处理"只是把问题搬家。

{{< sentence >}}
把一个问题彻底消灭，可能会顺手毁掉别的东西。
{{< /sentence >}}

不是"可能有副作用"这么温和。是你的胜利条件本身可能太窄了。

{{< sentence >}}
解决得太彻底，常常说明你把系统里别的功能也一起抹掉了。
{{< /sentence >}}

特别适合拿来防制度洁癖。凡是特别干净、特别一刀切的解法，都值得多问一次。

## 改题、停手和放弃

{{< sentence >}}
投入之后承认自己接错了题，比想出办法更难。
{{< /sentence >}}

这句直指最硬的一刀。许多人不是不会重命题，是前面已经投入了就不愿意回头。

{{< sentence >}}
不是所有问题都值得做到底。
{{< /sentence >}}

和"负责到底"有张力。现实里有些题目该移交，有些该暴露，有些只该先止血。

{{< sentence >}}
有些问题最好的处理，不是修好，而是停在代价可接受的位置。
{{< /sentence >}}

知道什么时候收手，本身就是问题处理能力的一部分。
