# 项目管理靠判断不靠流程——Berkun 反复在说的几句话

项目出问题的时候，翻工具手册没用，翻流程文档也没用。下面这些提醒句，是在项目经理最容易判断错的地方设下的锚。

## 关于项目经理到底在做什么

{{< sentence >}}
项目管理不是一门科学——它是把一件事从想法带到现实的实践，而这个过程从来都是混乱的。
{{< /sentence >}}

你以为有了方法论就能把混乱消灭。Berkun 说混乱是常态，方法论帮你在混乱中做出能接受的判断，不帮你消灭混乱本身。下次项目乱的时候，先检查自己是在对抗混乱，还是在混乱里做判断。

{{< sentence >}}
项目经理真正的工作是做决定。排期、文档、会议都只是做决定的手段。
{{< /sentence >}}

如果你的一天全花在更新排期表和写周报上，很可能没人在做真正的决定。这些工具应该为判断服务。当工具本身变成了目的，判断就停了。

## 关于计划与不确定性

{{< sentence >}}
排期表的价值不在于预测未来，而在于迫使你思考未来。
{{< /sentence >}}

排期表填完那一刻就开始过时。它的作用不是锁定结果，是让团队对"下一步做什么"有一个可以讨论的起点。

死守排期表不是尊重计划，是拒绝判断。

{{< sentence >}}
好的规格说明书让人在没有你在场的时候也能做出对的决定。
{{< /sentence >}}

写需求文档不是为了证明你想过了，是为了让别人不用再猜你想的是什么。如果文档写完之后，工程师还是每天来找你问"这个到底是什么意思"——那文档没有完成它的工作。

{{< sentence >}}
当你发现每个人对同一件事的理解都不一样时，你不是在浪费时间——你正在做项目管理中最重要的那件事。
{{< /sentence >}}

对齐理解是项目经理的核心输出。当所有人觉得"我们都清楚了"，项目经理该做的事恰好是去检验这个"清楚"到底是不是真的。

## 关于冲突与决策

{{< sentence >}}
如果一个决定所有人都同意，要么这个决定太简单，要么有人没说真话。
{{< /sentence >}}

项目里的全票通过经常是警告信号。有人不想惹麻烦，有人觉得说了也没用，有人根本没在听。Berkun 不怕分歧，怕假共识。

{{< sentence >}}
冲突是不可避免的。你不解决冲突，冲突会解决你的项目。
{{< /sentence >}}

绕开冲突是项目经理最贵的习惯。每一个被绕开的分歧，三周后都会变成一个更大的问题，而且到那时候修复成本翻倍。

## 关于权力和推动

{{< sentence >}}
权威不是别人给你的，是你在推动事情的过程中慢慢挣来的。
{{< /sentence >}}

新项目经理最常犯的错是等着被授权。Berkun 的经验是：你不需要等组织架构图给你一个头衔；你把一件卡住的事推动了，权威自然就来了。

{{< sentence >}}
说'不'是项目经理最重要的能力之一——但大部分项目经理说不出口。
{{< /sentence >}}

需求膨胀的根源很少是"需求太多"，通常是"没人敢说不"。每接一个新需求，都应该先问：这件事挤掉的是什么？如果答不出来，就是在假装资源是无限的。

## 什么场景拿出来用

项目排期又改了，团队开始抱怨——翻第 3、4 句。

会议上所有人都说"没问题"，但你感觉哪里不对——翻第 6 句。

两个人有分歧你想等等看——翻第 7 句。

老板又加需求你不知道怎么接——翻第 9 句。

你觉得自己每天都在忙但项目没进展——翻第 2 句，问问自己今天做了几个决定。
