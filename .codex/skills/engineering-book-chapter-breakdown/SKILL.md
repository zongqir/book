# 逐章拆解

## §1 命中判断

用户要求逐章拆解一本书时启用。

典型触发：
- "帮我逐章拆这本书"
- "先做前三章"
- "把第 N 章拆出来"
- "这章写得像摘要，帮我重写"

### 什么时候值得逐章

逐章是最深的阅读方式。不是每本书都值得逐章。值得的情况：
- 每章有独立的判断价值，不只是为全书论证服务
- 章节之间有递进关系，跳着读会漏掉东西
- 你打算把某些章当参考手册用——以后遇到具体问题回来查
- 书足够密，单靠 00-10 槽位压不住

逐章和固定槽位（00-10）独立并存：槽位做跨章综合，逐章做按章说透。两者不是替代关系。

### 书的类型不限于工程书

默认对工程、技术、系统类书籍最有效。非技术书逐章时，把"工程判断"替换为该书的核心价值类型：推理链、方法链、论证链、决策框架等。核心写法规则（说透、内容标题、不元解释、保留锋利内容）不因书的类型改变。

## §2 元数据

通用必填字段遵循 `AGENTS.md` base schema。

扩展字段：
- `audience`：最适合读这章拆解的人
- `chapter_no`：章号
- `chapter_basis`：原书章名或目录原文

## §3 工作流

### §3.1 先判断这章到底在说什么

动笔前回答四个问题：
- 这章到底在帮人理解或判断什么
- 读完这章以后，读者最该少犯什么错
- 如果只能带走一件事，这章最该留下什么
- 这章和前后章的关系是什么——为什么放在这个位置

如果这章偏背景或铺垫，也要逼自己回答：这份背景到底会改变后面哪一步判断或理解。

### §3.2 说透——展开到读者不用倒回去重读

"说透"是逐章的核心目标。你已经选了逐章而不是摘要，就意味着每章要展开到读者真正理解为止。

说透的操作性检验——读者读完每一段，能回答这四个问题：

1. **看什么**：这一步要观察什么指标、什么现象、什么变化
2. **怎么看**：用什么工具、什么命令、看哪个字段
3. **数字意味着什么**：结果高/低/异常分别说明什么，正常范围大概是什么
4. **下一步怎么走**：根据这一步的结果，接下来往哪个方向继续

如果读完一段，读者还要去翻原书才能动手，就没说透。

这四条对工程/技术书最直接。非技术书逐章时，替换成对应的检验："判断是什么→依据是什么→边界在哪→和下一步什么关系"。

说透不是说长。100 行说透比 200 行说不透更好。关键是每句话都在推进理解，不在重复、铺垫或自我解释。

### §3.3 保留原书最锋利的内容

写逐章拆解时，最容易犯的错是把原书最具体、最反直觉的内容抽象化。

先主动找这几类内容：
- 会改变现场动作的顺序反转
- 会改变层级判断的时间量级或数字直觉
- 会改变指标解释的语义区别
- 具体的入口、工具、命令、队列名

这些内容一旦存在，必须保留原始精度，不能先压成泛化方法论。

检验：如果改写后，原书里"几毫秒已经不是 CPU 世界"变成了"先看证据层级"，你就把最值钱的部分磨平了。

### §3.4 标题说内容，先给答案

**标题**：直接说这段的判断、误判或动作。

写成：
- `CPU 高，先分清是在做工还是在排队`
- `热点出来以后，还不能宣布根因`
- `忙碌不等于已经形成约束`

不要写成：
- `本章定位` `问题驱动` `工程价值驱动`
- `这章真正回答了什么` `这章最值钱的地方是`
- `手里先留这三条资产` `最小沉淀`

检验：标题换到别的章节也成立 → 还是结构壳。标题没有具体对象 → 还不够贴内容。

**正文**：先给判断，再补原因和动作。不要在结论前铺一层"我现在来解释这章在干什么"。

不要写：
- `这章真正想让你分开的，不是……而是……`
- `如果把这章只读成……会漏掉……`
- `作者真正想说的是……`
- `这张卡解决的是……`

这些句子在讲"讲法"，不在讲"内容"。删掉前半句，直接写后面的判断。

### §3.5 memory-card shortcode

每章结尾用 `{{</* memory-card */>}}` 留一张记忆卡。四个字段：

```
{{</* memory-card
  chapter="记忆卡 06"
  title="CPU 高，先分做工、排队、降频和失真"
  phenomenon="看到什么现象"
  dont="先别急着下什么结论"
  check="先补哪组证据"
  upgrade="什么时候才能升级成结论"
*/>}}
```

memory-card 是这章最浓缩的判断锚点。写的时候问自己：如果读者三个月后只记得这张卡，够不够用。

### §3.6 章间衔接

每章结尾用 1-2 句自然连接到下一章。说清楚"这章解决了什么，下一章接着解决什么"。

不要写成目录预告（"接下来第 N 章会讲……"），而是写成推理的自然递进（"CPU 分清以后，再看内存是不是也在骗人"）。

### §3.7 反模式黑名单

以下表述不进正文：

- `这章真正想/要/在……` `作者真正……` `这里真正……`
- `这章最值钱的地方是` `这章最重要的提醒是`
- `本章定位` `问题驱动` `工程价值驱动` `本章最小沉淀`
- `如果把这章只读成……` `为什么这章老让人读成……`
- `换句话说` `简单来说` 作为句子开头
- `最后压成三张卡` `留一张系统路径图` 但不展开内容
- `第 N 章主要拦住的是` `这张卡解决的是`
- 高频 `不是……而是……`（偶尔可以，一页超过 3 次就太多）

凡是写到"卡""清单""协议""模板""图"，正文里必须展开具体内容，不能只报产物名。

### §3.8 篇幅

**判断标准是说透没有，不是行数。**

说透了 80 行也够，没说透 300 行也不够。参考范围：背景铺垫章 80-120 行，普通章节 120-200 行，密度特别高的章节（CPU、网络、内存）可以到 300 行。

检验标准见 §3.2 的四条操作性检验。

## §4 终检清单

提交前逐条过：

1. 标题是内容句（判断、误判或动作），不是结构壳
2. 每个 H2 换到别的章节不能也成立
3. 正文先给判断再补原因，没有先解释"我在讲什么"
4. 没有"这章真正……""作者真正……"式元解释
5. 原书最锋利的内容（顺序反转、量级、语义区别、具体入口）保住了
6. 写到卡/清单/图/协议的地方都展开了具体内容
7. 说透了——读者不用倒回原书就能理解这章
8. 误判和为什么容易犯都写清了
9. 章间衔接自然，不是目录预告
10. memory-card shortcode 在结尾，四字段填完
11. "不是……而是……"一页不超过 3 次
12. 没有高频"真正""换句话说""简单来说"
13. 没有评价句或重要性定级句
14. 没有自我指涉（"这页""这篇拆解"）
15. 没有导游词或过程叙述
16. 没有生造复合词
17. 第一次读能直接懂，不用倒回去
18. frontmatter 完整，`chapter_no` 和 `chapter_basis` 填了
19. `evidence_status` 准确
20. 时间字段写昨天

## 输出规则

- 说透优先于精简——逐章不是摘要，但也不是注水
- 先给判断，不先解释你在组织什么
- 保留锋利内容的原始精度，不抽象化
- 写到卡/图/清单必须当场展开
- 每章结尾附 memory-card shortcode
- 非技术书逐章时，把"工程判断"换成该书的核心价值类型

## 校准与例外

- 如果一本书只有 3-5 章且每章不厚，逐章可能不如直接用 00-10 槽位
- 纯理论章或纯背景章可以写短（80 行），但仍然要回答"这份背景改变了后面什么判断"
- 如果原书一章里塞了两个完全独立的主题，可以拆成两篇
- `00_逐章问题地图` 是可选的协调文件，不是每本书都需要
