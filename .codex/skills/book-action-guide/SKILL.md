# 行动指南

把 `04_行动指南.md` 当成一份让读者"现在就能开始做，并在过程中知道何时继续、何时调整、何时算完成"的执行稿。

## 核心原则

一份行动指南需要覆盖这些功能维度。它们是内容清单，不是 H2 标题——最终的文章结构由这本书的行动逻辑决定：

**⚠️ 以下维度是思考工具，不是输出结构。**

它们帮你想清楚要写什么，但不决定你的 H2 标题、段落顺序或文章结构。最终结构必须从这本书的内容中自然长出来。写完后检查：如果你的 H2 标题序列和同分类下另一本书的相同槽位一样，说明你把思考维度当成了输出模板——必须重新组织。

1. **启动时机**：什么时候该打开这份指南
2. **最小闭环**：第一次就能跑通的最短路径
3. **标准流程**：完整版动作序列，判断点嵌在里面
4. **调整策略**：做偏了或遇到特殊材料怎么办
5. **产物与终点**：跑完后留下什么、做到哪算完
6. **一页清单**：压成一页的检查表

H2 标题不要直接用上面的功能名。"什么时候启动""最小产物与完成标准"这类写法是结构壳——和"案例一""第一步"一样，告诉读者的是节的角色，不是节的内容。标题从这本书的内容里长出来，读者光看标题就该知道这一块在讲什么。

同样不应该出现这些额外章节："总原则""使用前提""不要这样用"。总序列压成开头一两句话，前提放元数据 `prerequisites`，边界嵌进步骤或调整段。

判断点、升级条件、失败信号、回退条件这些内容通常不单独立章节，优先嵌进标准流程里。

## 1. 先判定是否命中

出现下面这些情况，就按这个 skill 处理：
- 用户直接提到 `04_行动指南`、`04` 文件、行动指南
- 用户要新建、重写、改写、润色、审阅行动指南文稿
- 用户说行动指南写得像方法论、缺判断点、只有步骤没有产物和完成标准
- 任务语境明确在读书库或研究资料库里处理执行类文档

下面这些任务不要落到 `04`：
- 讲为什么读这本书：去 `00_为什么读这本书.md`
- 讲调用场景和误判：去 `03_最有价值的案例.md`
- 讲方法框架和部件关系：去 `05_方法论总结.md`
- 讲适用前提、不要硬套、失效信号和停退换判断：去 `06_适用边界与失效条件.md`
- 只想展开跨场景共性失败模式：只有材料特别厚时，再去 `07_反例与失败条件.md`

如果一段主要在讲"它为什么重要"或"它的方法多完整"，通常就写偏了。

## 2. 先补元数据

通用字段与语义统一遵循仓库 `AGENTS.md`。这个槽位的扩展字段：`action_scope`、`default_horizon`、`prerequisites`、`activation_signals`、`start_state`、`target_state`。

```yaml
action_scope: "单本书的一轮阅读闭环"
default_horizon: "1–7 天"
prerequisites: "已有一本准备阅读的书；愿意读前和读后各留至少一版记录。"
```

同时先把这两个通用字段写对：
- `title` 不要直接写 `行动指南`，而要把这套执行流程最核心的动作判断压成一句内容标题
- `summary` 直接概括读者照着做会得到什么，不要写成 `把某书压成一套执行流程` 这类任务说明

## 3. 工作流

### NotebookLM 提问约束

当你不确定一本书到底能不能支撑 `04`，或担心会把方法框架硬写成执行稿时，先让 NotebookLM 按这个 skill 的标准回答槽位成立性。

- 先问成立性：原书里有没有稳定的动作顺序、判断点、最小闭环和完成边界？
- 再问筛选性：哪些动作是真正能跑通的执行线，哪些只是原则或方法说明？
- 最后问退出性：如果 `04` 立不住，是因为这本书只有观点和方法，没有足够执行闭环，还是因为 source 还不够？

优先问这种问题：

- 原书里最稳定的最小闭环是什么？
- 哪些内容应该进 `04`，哪些更应该留在 `05` 或 `06`？
- 如果只能先写最小版行动指南，最该保留哪 3 到 5 个动作和判断点？

不要问：`这本书怎么用？`

通用判定方法见 [../references/notebooklm-slot-gating.md](../references/notebooklm-slot-gating.md)。

### 先排动作顺序

不要直接照抄原书章节顺序。先把书里的关键动作压成一条执行线：
1. 找出最关键的 4–6 个动作
2. 排出先后顺序
3. 从里面抽出第一次就能跑通的最小闭环
4. 留下必要的标准流程

优先保留：能马上开始做的、顺序清楚的、做完后有明显结果的、能看出是否做对的。

### 把判断嵌进步骤

标准流程里最关键的不是把步骤写满，而是把判断写活。在关键步骤下面，按需直接补：
- 做到什么程度可以继续
- 什么信号说明做偏了
- 什么情况下回到前一步
- 什么情况下从最小版升级到标准版

不要为了"显得完整"把这些全部拆成独立章节。

### 最后再收口结果

写完步骤后，再回头补两件事：跑完后至少留下什么最小产物、做到什么程度才算真的完成而不只是"看起来做过"。

工具接入也在这一步考虑：默认先脱离工具跑出最小闭环，只有在核验、对照或持续推进时再接工具。

## 4. 终检清单

### 反同质化（必查）

- [ ] H2 标题序列和同分类下已有书的同槽位不同
- [ ] 开头第一句和同分类下已有书的开头不同
- [ ] 全篇对比句式（"不是X而是Y""与其A不如B"）合计不超过 2 次
- [ ] 没有使用可以套到任何书上的万能句式作为标题或开头

提交前逐条过一遍：

**核心**
读者读完以后，知道先做什么、怎么判断做得对不对、最后留下什么

**元数据**
- [ ] `title` 是内容判断，不是 `行动指南`
- [ ] `summary` 直接写读者照着做会得到什么，不是任务说明
- [ ] `created_at` / `updated_at` 是完整带时区的 ISO 时间
- [ ] `evidence_status` 与实际来源一致
- [ ] `action_scope`、`default_horizon`、`prerequisites` 已填写

**结构**
- [ ] 6 个功能都覆盖到了，没有遗漏
- [ ] 没有多出"总原则""使用前提""不要这样用"等额外章节
- [ ] H2 标题是内容判断，不是功能名（不是"什么时候启动""最小产物与完成标准"）
- [ ] 最小闭环真的能单独跑通，不只是标准流程的压缩版
- [ ] 判断点嵌在步骤里，不单独拆成标题
- [ ] 正文 H1 与 frontmatter `title` 一致

**执行感**
- [ ] 每一步都有明确动作，不只有"要注意""很重要"
- [ ] 判断点有触发信号，不只有"如果做偏了"
- [ ] 最小产物具体到能数出来（"3 个主张""1 段摘要"），不是模糊描述
- [ ] 完成标准能区分"真的完成"和"看起来做过"

**槽位边界**
- [ ] 没有写成方法论总结（→ 05）
- [ ] 没有混入调用场景和误判（→ 03）
- [ ] 没有混入边界条件（→ 06）
- [ ] 没有大段复述原书章节结构

**文风**
- [ ] 读起来像执行稿，不像说明书
- [ ] 没有连续出现"这本书告诉我们""作者指出"这类转述腔
- [ ] 段落有呼吸——不是每段都一样长、一样密
- [ ] 开头没有反向自我定义，结尾没有给文章写使用说明

**长度**
- [ ] 单页行动指南正文在 120–180 行之间；低于 120 行说明步骤或判断点没展开，超过 180 行检查是否混入了方法论内容

## 输出规则

- 优先让人照着做出来，不优先把道理讲满
- 优先最小闭环，不优先完整版框架
- 优先把判断点写进步骤，不优先把概念拆成栏目
- 6 个功能都覆盖，但 H2 标题从内容里来，不用功能名当标题
- 步骤标题写动作判断，不写"第一步""第二步"
- 文风保持执行感和对话感，避免说明书腔
- 改现有稿子时，保留真正能执行的段落，不做机械式全量重写
- 正文典型长度 120–180 行

## 校准与例外

- 如果内容主要在回答"它的方法框架是什么"，它更像 `05_方法论总结.md`。
- 如果内容主要在回答"什么场景最容易误判"，它更像 `03_最有价值的案例.md`。
- 如果只有步骤没有判断点，不是好的行动指南。
- 如果只有产物没有完成标准，完成边界还没画清。
- 如果材料不足以支撑完整行动指南，宁可只写最小闭环，也不要硬凑标准流程。
