{{< sentence >}}建模错了，算法再快也是在高效地解一个错的问题。{{< /sentence >}}

工程中最大的算法错误不是选了一个慢的算法，而是把问题建模错了。同一个业务需求可以建模成排序、搜索、图问题或优化问题——建模方式决定了解法空间。花 30 分钟想清楚问题是什么，比花三天优化一个方向错误的实现更值。

{{< sentence >}}当你觉得"再优化一下就行了"的时候，先确认这个问题不是 NP-hard。{{< /sentence >}}

NP-hard 问题在实际需求中经常伪装成"看起来不难"的样子。排课表、分配仓库、路径规划——产品经理描述起来都是一句话的事情。但如果它本质上是旅行商问题或集合覆盖问题，不存在"再优化一下"的空间。识别出 NP-hard 是止损，不是放弃。

{{< sentence >}}在用花哨算法之前，先试排序。{{< /sentence >}}

大量看起来需要复杂算法的问题，排序之后变得简单。找中位数、去重、合并区间、找最近邻——排序完成后这些问题都变成了线性扫描。O(n log n) 的排序加 O(n) 的扫描，通常比直接用 O(n²) 的暴力法快，而且代码简单得多。

{{< sentence >}}暴力解法不丢人，但你得知道它的复杂度。{{< /sentence >}}

写出暴力解法是解决任何算法问题的第一步。它的价值不在于"能跑"，在于提供一个基线：你知道当前解法有多慢，才能判断优化值不值得。如果暴力解法在实际数据量下已经够快（100ms 以内），优化到 10ms 的 ROI 可能不高。

{{< sentence >}}渐近复杂度不是一切。常数因子和缓存友好性在实际性能中经常是决定性的。{{< /sentence >}}

O(n log n) 和 O(n) 在 n=1000 的时候差别不大。但一个缓存友好的 O(n log n) 算法可能比一个频繁 cache miss 的 O(n) 算法快三倍。Skiena 反复强调：理论分析给出方向，实测给出答案。不要只看大 O，要跑 benchmark。

{{< sentence >}}图是最通用的建模工具。不确定问题怎么建模的时候，先画成图。{{< /sentence >}}

社交关系、依赖管理、调度约束、网络路由——大量看似不相关的问题画成图之后都变成了经典图算法问题。连通性用 BFS，最短路径用 Dijkstra，依赖排序用拓扑排序，最小覆盖用网络流。图模型是问题建模的瑞士军刀。

{{< sentence >}}动态规划不是一种算法，是一种思考方式：把大问题拆成重叠子问题，记住子问题的答案。{{< /sentence >}}

很多人觉得动态规划很难，是因为把它当成一种需要"灵光一现"的技巧。Skiena 的讲法不同：DP 的核心只有两步——定义子问题的状态，写出状态转移方程。难点在第一步：找到合适的状态定义。这需要练习，但不需要天赋。
